# 신규 기획서 vs 현재 코드베이스 분석 리포트

> 작성일: 2026-04-03
> 대상: `/Downloads/grapit.md` (기획자 신규 기획서) vs 현재 Grapit 프로젝트

---

## 요약

신규 기획서는 **서비스 정체성 자체를 재정의**한다. 현재 프로젝트가 "범용 공연 티켓 예매 플랫폼(인터파크 클론)"으로 구축된 반면, 신규 기획서는 **"팬-창작자 이벤트 예약 전문 플랫폼"**으로 포지셔닝한다. 이로 인해 카테고리 체계, 네비게이션, 예매 플로우, 상태 머신 등 거의 모든 영역에서 변경이 필요하다.

아래에서 영역별로 **현재 상태 → 기획서 요구사항 → 영향도 → 논의 사항**을 정리한다.

---

## 1. 서비스 정체성 & 포지셔닝 [CRITICAL - 근본적 변경]

### 현재
- 비전: "모든 라이브 엔터테인먼트를 하나의 플랫폼에서 발견하고 예매"
- 포지셔닝: NOL 티켓(인터파크) 참조 → 범용 공연 티켓
- 브랜드: **Grapit**

### 기획서
- 비전: "팬이 좋아하는 창작자·출연자와 직접 만나는 모든 이벤트"
- 포지셔닝: 인터파크(대형 공연) / 윗치폼(굿즈) 사이의 **팬-창작자 이벤트 전문** 틈새
- 브랜드: **Grab it**
- 차별점: 추첨 자동화, 소규모(30~500명) 셀프서브 이벤트, 온라인 팝업 입장권

### 논의 필요
- [ ] **브랜드명 확정**: "Grapit" vs "Grab it" — 코드베이스, 도메인, 패키지명 전체 영향
- [ ] **타겟 사용자 전면 교체**: 기존 3 Persona(공연매니아/가족관람/K-POP팬)를 폐기하고, 팬미팅·팬사인회·팝업 참가자 중심으로 재정의해야 함
- [ ] **"공연(Performance)" 용어를 "이벤트(Event)"로 변경할지** — DB 테이블명 `performances` → `events`, API 경로 `/performances` → `/events` 등 전면 리네이밍 이슈

## 2. 카테고리 체계 [CRITICAL - 전면 재구축]

### 현재
8개 장르 기반:
```
뮤지컬 / 콘서트 / 스포츠 / 전시·행사 / 클래식·무용 / 아동·가족 / 연극 / 레저·캠핑
```
- DB: `performances.genre` enum
- 라우트: `/genre/[genre]`

### 기획서
IP 5대 분류 체계:
```
아티스트·셀럽 / 창작자 / IP 팝업 / 버추얼 / 인플루언서
```
각 대분류에 서브분류 존재 (GL·BL 출연자, 웹툰 작가, 버추얼 유튜버 등)

### 영향도: 매우 높음
- `performances.genre` enum 전면 교체
- `/genre/[genre]` 라우트 → `/category/[category]` 등으로 변경
- 홈페이지, 검색, 필터, GNB 모두 영향
- 기존 테스트 데이터·시드 전부 교체

### 논의 필요
- [ ] Phase별 분류 출시: P1(아티스트·셀럽, 창작자, IP 팝업), P2(일부 확장), P3(버추얼, 인플루언서) — **MVP에서 P1 3개만 구현하고 나머지는 비활성?**
- [ ] "IP 태그" 개념 신규 도입 필요 — 현재 DB에 태그 테이블 없음
- [ ] 서브분류를 DB에 어떻게 모델링할 것인가 (별도 테이블 vs JSON vs enum)

## 3. GNB & 네비게이션 [HIGH - 구조 변경]

### 현재
- GNB: 로고 / 장르별 네비게이션 (8개) / 검색 / 로그인·마이페이지
- 컴포넌트: `components/layout/GNB.tsx`

### 기획서
- **3탭 구조**: 홈 / 티켓팅 / 팝업
- 티켓팅 탭: IP분류·예약방식·인기·마감임박·오픈예정·지역 필터
- 팝업 탭: 온라인/오프라인 서브필터, IP분류, 상태 필터

### 영향도: 높음
- GNB 컴포넌트 전면 재설계
- 라우팅 구조 변경 (새 탭별 페이지 필요)
- 기존 `/genre/[genre]` 라우트 폐기 또는 재활용

### 논의 필요
- [ ] 모바일에서 3탭 = 하단 탭바? 아니면 상단 탭?
- [ ] "팝업" 탭이 MVP에 필요한가? — 온라인 팝업 시스템은 완전 신규 기능

## 4. 이벤트 상태 시스템 [HIGH - 신규 상태 머신]

### 현재
- `performances.status`: 단순 상태 (판매중, 판매종료 등 — 정확한 enum은 코드 확인 필요)
- `reservations.status`: `PENDING_PAYMENT`, `CONFIRMED`, `CANCELLED`, `FAILED`

### 기획서
**6단계 이벤트 상태:**
| 상태 | CTA |
|------|-----|
| 추후공지 | 알림 신청 |
| 오픈 예정 | 알림 신청 |
| 선착순 판매 중 | 예매하기 |
| 추첨 응모 중 | 응모하기 |
| 마감 | 비활성 |
| 취소됨 | 비활성 |

**추가 배지**: HOT, 단독판매
**오픈 시점 표시 규칙**: D-N 방식 사용 금지 → "오늘 HH:MM" / "내일 HH:MM" / "MM.DD(요일) HH:MM"

### 영향도: 높음
- `performances.status` enum 교체 + 전환 로직 구현
- 이벤트 카드 컴포넌트 전면 수정 (상태별 배지·CTA 분기)
- 오픈 시점 표시 포맷터 유틸리티 신규 구현

### 논의 필요
- [ ] "추첨 응모 중"은 추첨 시스템이 구현된 후에만 의미 있음 — MVP 범위?
- [ ] 상태 전환 자동화: 오픈 일시 도래 시 자동으로 "오픈 예정" → "판매 중" 전환? → pg-boss 잡으로 구현?
- [ ] "추후공지" 상태의 이벤트는 어떻게 생성? 오픈 일시 없이 생성 허용?

## 5. 이벤트 상세 페이지 [HIGH - 레이아웃 재설계]

### 현재
- 단일 컬럼 레이아웃 + 탭(정보, 캐스팅)
- `/performance/[id]` 페이지

### 기획서
- **3열 레이아웃**: 좌측(포스터·이벤트명·태그) / 중앙(기본정보 + 4탭) / 우측(상태별 CTA 패널)
- **4탭**: 이벤트 정보 / 캐스팅 정보 / 판매자 정보 / Q&A
- **하단 공통**: 예매자 통계 (성별·연령대)

### 영향도: 높음 — 페이지 전면 재작성
- 3열 반응형 레이아웃 신규 구현
- 우측 CTA 패널: 이벤트 상태별 6가지 분기 렌더링
- **판매자 정보 탭**: 새 DB 테이블 필요 (판매자/기획사 정보)
- **Q&A 탭**: 완전 신규 기능 (질문·답변·신고)
- **예매자 통계**: 성별·연령대 집계 API 신규

### 논의 필요
- [ ] Q&A 시스템은 MVP에 포함? — 백엔드(질문 테이블, 답변 CRUD) + 프론트 모두 신규
- [ ] 판매자 정보: Phase 1에서는 (주)아이콘스만 운영 → 하드코딩 가능? 아니면 DB 설계?
- [ ] 예매자 통계: 사용자 정보(성별·생년월일)가 필수 입력이어야 함 — 현재 `users.gender`, `users.birthDate`는 있으나 필수가 아닐 수 있음

## 6. 예매 플로우 [CRITICAL - 4개 분기 플로우]

### 현재
**단일 플로우:**
```
이벤트 상세 → 예매하기 → 날짜·회차 선택 → 좌석 선택 → 결제 확인 → 결제 → 완료
```
- 유료 선착순만 지원
- 좌석 선택 필수 (SVG 좌석맵)

### 기획서
**6개 플로우:**
| 유형 | 핵심 차이 |
|------|----------|
| 선착순 유료 | 대기열 → 좌석·등급·수량 → 예매 안내 모달 → 결제 |
| 선착순 무료 | 대기열 → 등급·수량 → 신청 완료 (결제 없음) |
| 추첨 유료 | 등급·수량 → 응모 → 추첨 → 당첨 알림 → 기한 내 결제 |
| 추첨 무료 | 응모 → 추첨 → 당첨 자동 확정 |
| 온라인 팝업 | 선착순 신청 → 링크+비밀번호 발송 → 비밀번호 입장 |
| 오프라인 팝업 | 방문일·시간대·인원 → QR 발급 |

### 영향도: 매우 높음 — 아키텍처 수준 변경
현재 구현된 단일 플로우를 기반으로 5개 추가 분기가 필요하며, 각각 별도의 UI·API·상태 관리가 요구됨.

### 신규 시스템 필요 목록
| 시스템 | 현재 상태 | 필요 작업량 |
|--------|----------|-----------|
| 대기열 (Queue) | 미구현 | **대규모** — Redis Sorted Set + WebSocket + UI |
| 추첨 (Lottery) | 미구현 | **대규모** — 응모 DB, 추첨 엔진, 당첨 알림, 결제 기한 |
| 무료 이벤트 처리 | 미구현 | 중간 — 결제 스킵 로직, 상태 분기 |
| 온라인 팝업 입장 | 미구현 | **대규모** — 비밀번호 시스템, IP 컨셉 페이지, 보안 |
| 오프라인 팝업 QR | 미구현 | 중간 — QR 생성, 스캐너 페이지 |
| 예매 안내 모달 | 미구현 | 소규모 — 주최자 입력 리치텍스트 표시 |

### 논의 필요
- [ ] **MVP 범위 결정이 시급**: 6개 플로우를 모두 MVP에 넣으면 개발 기간이 수배로 늘어남
- [ ] **추천 MVP**: "선착순 유료" + "선착순 무료" 만 먼저 구현, 추첨·팝업은 Phase 2로
- [ ] 대기열: 트래픽이 적은 초기에 정말 필요한가? → 일정 임계치 초과 시에만 활성화하는 방식?
- [ ] 현재 홀드 타이머 10분 → 기획서 7~10분 — **설정 가능하게** 변경할 것인가?

## 7. 대기열 시스템 [NEW - 완전 신규]

### 기획서 요구사항
- 선착순 이벤트에서 고인기 오픈 시 활성화
- **1계정 3기기까지 허용**, 4기기 이상 초과 기기 토큰 무효화
- 홀드: 유료 선착순 7~10분
- UI: 현재 순번 / 진행 바 / 예상 대기시간 / 잔여 좌석

### 기술적 고려
- Redis Sorted Set (타임스탬프 기반 순번 관리)
- WebSocket으로 순번 업데이트 실시간 푸시
- 기기 추적: JWT + device fingerprint 또는 세션별 기기 ID
- pg-boss로 대기열 만료·정리 잡

### 논의 필요
- [ ] 1계정 3기기 정책: 어떻게 "기기"를 식별? (User-Agent? fingerprint? 별도 device token?)
- [ ] 대기열 활성화 기준: 자동(동시접속 N명 초과)? 수동(어드민 설정)?
- [ ] MVP에서 대기열 없이 론칭 가능한가?

## 8. 추첨 시스템 [NEW - 완전 신규]

### 기획서 요구사항
- 응모 기간 설정 → 마감 후 추첨 실행
- 공정성 증명 로그 영구 보관
- 당첨 발표 공개 URL 자동 생성
- 당첨자 결제 기한 (72시간~), 미결제 시 자동 취소 → 차순위 당첨
- 어드민: 추첨 설정·실행·당첨자 관리

### 필요한 DB 테이블 (신규)
```
applications (응모)
  - id, eventId, userId, showtimeId, tierName, quantity, applicationNumber, status, createdAt

lottery_results (추첨 결과)
  - id, eventId, executedAt, fairnessLog, publicUrl

lottery_winners (당첨자)
  - id, applicationId, rank, paymentDeadline, status (PENDING/PAID/EXPIRED/CANCELLED)
```

### 논의 필요
- [ ] 추첨 알고리즘: 단순 무작위(crypto.randomBytes) vs 가중치(VIP 우선)?
- [ ] 공정성 증명: 블록체인 해시? 감사 로그? 재현 가능 시드?
- [ ] MVP에 추첨 포함할 것인지 — **기획서가 "Phase 1 기능"으로 추첨 관리를 포함**시킨 점 주목

## 9. 온라인 팝업 시스템 [NEW - 완전 신규]

### 기획서 요구사항
- 결제 없는 선착순 입장 예약
- 링크 + 비밀번호 이메일/우편 발송
- 비밀번호 입력 → IP 컨셉 페이지 접근
- 오류 3회 시 1시간 잠금
- 입장 기간 만료 시 접근 차단

### 논의 필요
- [ ] "IP 컨셉 페이지"는 주최자가 직접 꾸미는 랜딩 페이지? → CMS/페이지 빌더 필요?
- [ ] 비밀번호 발송 수단: 이메일은 가능하나, "우편" 발송은 물리적 배송 — 시스템 범위 밖?
- [ ] 한정 굿즈 구매 연동: 쇼핑몰 URL 외부 링크? 자체 구매 시스템?
- [ ] **이 기능은 독립 서비스 수준의 복잡도** — MVP에서 제외 강력 권장

## 10. 취소·환불 정책 [MEDIUM - 정책 변경]

### 현재
- 단순: 공연 시작 24시간 전까지 취소 가능
- `reservations.cancelDeadline` = showtime - 24h

### 기획서
**소비자분쟁해결기준 기반 단계별 환불:**
| 취소 시점 | 환불율 |
|----------|--------|
| 10일 전 | 100% |
| 7일 전 | 90% |
| 3일 전 | 80% |
| 1일 전 | 70% |
| 당일/이후 | 불가 |

**무료 이벤트**: 3시간 전까지 취소 (환불 없음)
**주최자 귀책**: 전액 + 10% 배상

### 영향도: 중간
- 환불 금액 계산 로직 수정 (현재 전액 or 불가 → 단계별 비율)
- 프론트엔드: 취소 모달에 현재 시점 기준 환불 가능 금액 표시
- DB: `payments` 테이블에 `refundAmount`, `deductionRate` 필드 추가 고려

### 논의 필요
- [ ] 현재 Toss Payments `cancel` API가 부분 환불을 지원하는지 확인 필요
- [ ] 환불 비율을 이벤트별로 주최자가 커스터마이징할 수 있는지, 아니면 플랫폼 고정 정책인지

## 11. 마이페이지 [HIGH - 대폭 확장]

### A. 즉시 합의 필요 사항 (기획자와 논의)
