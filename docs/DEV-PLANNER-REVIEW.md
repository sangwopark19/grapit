## 1. 요약

### 현재 상태

인증, 공연 카탈로그, SVG 좌석 선택, Toss Payments 결제, 기본 어드민을 갖춘 **범용 공연 티켓 예매 플랫폼**이 Cloud Run에 배포되어 동작 중입니다 (TypeScript 23,500+ LOC, 331 commits).

### 기획서 대비 완성도

| 지표 | 기획서 | 현재 구현 | 비율 |
|---|---|---|---|
| 사용자 화면 수 | ~25개 (S-01~S-15 + 마이페이지 6섹션) | 10개 | 40% |
| 예약 방식 | 6종 | 1종 (선착순 유료) | 17% |
| 어드민 섹션 | 12섹션 (대시보드 포함) | 3섹션 | 25% |
| 마이페이지 섹션 | 6섹션 | 1섹션 | 17% |
| 백엔드 시스템 | ~15개 핵심 시스템 | 6개 | 40% |
| RBAC 역할 | 6개 | 2개 (user/admin) | 33% |

### 핵심 이슈

**제품 정체성이 다릅니다.** 기획서는 "팬-크리에이터 이벤트 예약 전문 플랫폼"을 정의하고 있으나, 현재 구현은 "인터파크 스타일 범용 공연 티켓 예매 플랫폼"으로 구축되어 있습니다. 이 차이는 데이터 모델, 네비게이션 구조, 이벤트 분류 체계 전반에 영향을 미칩니다.

**개발 진행을 위해 12개 차단 질문(섹션 5)에 대한 기획자 답변이 필요합니다.**

## 2. 컨셉 정렬 (CRITICAL)

### 기획서 vs 현재 구현 비교

| 차원 | 기획서 (Grab it_MVP) | 현재 구현 | 갭 수준 |
|---|---|---|---|
| **제품 정체성** | 팬-크리에이터 이벤트 예약 전문 플랫폼 | 범용 공연 티켓 예매 플랫폼 | CRITICAL |
| **타겟 이벤트** | 팬미팅, 팬사인회, 팝업스토어 (30~500명 중소규모) | 뮤지컬, 콘서트, 연극, 전시 (대규모) | CRITICAL |
| **카테고리 체계** | IP 기반 5분류 (아티스트/셀럽, 창작자, IP 팝업, 버추얼, 인플루언서) | 장르 기반 8분류 (musical, concert, play, exhibition, classic, sports, kids_family, leisure_camping) | CRITICAL |
| **예약 방식** | 6종 (선착순 유/무료, 추첨 유/무료, 온라인 팝업, 오프라인 팝업) | 1종 (선착순 유료) | CRITICAL |
| **GNB 구조** | 4탭 (홈 / 티켓팅 / 팝업 / 오픈예정) + Zone B/C 필터 시스템 | 장르별 탭 네비게이션 | CRITICAL |
| **이벤트 상태** | 6단계 (추후공지, 오픈예정, 선착순 판매중, 추첨 응모중, 마감, 취소됨) | 4단계 (upcoming, selling, closing_soon, ended) | HIGH |
| **예약 상태 흐름** | 7단계 (예매완료, 예매확정, 응모중, 당첨/결제대기, 입장가능, 사용완료, 취소/낙첨) | 4단계 (PENDING_PAYMENT, CONFIRMED, CANCELLED, FAILED) | HIGH |
| **할인 체계** | 3단계 스택 (이벤트 할인 → 쿠폰 → 포인트) | 없음 | HIGH |
| **운영 모델** | (주)아이콘스 내부 운영팀 전용 (V1) | 관리자 1명 (admin 역할) | MEDIUM |

### 비교 설명

**카테고리 체계**: 기획서는 "어떤 IP/인물의 이벤트인가"를 기준으로 분류합니다 (GL/BL 출연자, 웹툰 작가, 캐릭터 IP 등). 현재 구현은 "어떤 장르의 공연인가"를 기준으로 분류합니다. 이 차이는 DB 스키마의 `genre` enum, 프론트엔드의 필터/검색 로직, GNB 구조 전체에 영향을 줍니다.

**GNB 구조**: 기획서는 "홈/티켓팅/팝업/오픈예정" 4탭에 각각 히어로 슬라이더, 필터 칩 바(Zone B), 정렬+이벤트 수 라인(Zone C)을 가진 통일된 레이아웃을 정의합니다. 현재 구현은 장르별 페이지(`/genre/[genre]`)로 네비게이션되며, 기획서의 탭 구조와 일치하지 않습니다.

**예약 방식**: 기획서의 6가지 예약 방식은 각각 완전히 다른 유저 플로우와 백엔드 로직을 요구합니다. 특히 추첨 시스템(응모 → 추첨 실행 → 당첨/낙첨 알림 → 당첨자 결제)과 팝업 입장 시스템(링크+비밀번호 발송, IP 테마 페이지)은 완전히 새로운 시스템입니다.

## 4. 기능 GAP 상세 목록

> 기획자는 각 GAP의 **V1 포함 여부**를 결정해주세요. 섹션 9 응답 템플릿에서 일괄 답변 가능합니다.

### 카테고리 범례

- **SCHEMA**: DB 테이블 변경 필요
- **BACKEND**: 새 API 엔드포인트/서비스 로직
- **FRONTEND**: 새 페이지 또는 대규모 UI 변경
- **FULL-STACK**: 프론트+백+DB 모두 변경
- **INFRA**: 인프라 또는 3rd party 연동

### 심각도 범례

- **CRITICAL**: 제품 정체성에 영향. 해결 없이 진행 불가.
- **HIGH**: 핵심 유저 플로우에 영향.
- **MEDIUM**: 운영 편의 또는 부가 기능.
- **LOW**: 있으면 좋으나 MVP 없이도 런칭 가능.

|---|---|---|---|---|---|---|
| GAP-01 | SCHEMA | 핵심 데이터 모델 전환 | `events` 테이블 + IP 카테고리 5분류 + 예약방식 6종 | `performances` 테이블 + 장르 8분류 + 예약방식 1종 | CRITICAL | 2~3주 |

현재 코드베이스는 `02-PRD.md`(인터파크 스타일 범용 공연 티켓 예매) 기준으로 구축되었습니다. `Grab it_MVP` 기획서는 "팬-크리에이터 이벤트 예약 전문 플랫폼"을 정의합니다. **어느 문서가 최종 권위 문서인가요?** `Grab it_MVP` 기획서 방향으로 전환합니까?

**BQ-02: 데이터 모델 재구조화**

전환한다면, DB의 `performances` 테이블 → `events` 테이블로 리네이밍하고, `genre` enum을 IP 카테고리 기반으로 변경해야 합니다. 이 작업은 **모든 API 엔드포인트, 프론트엔드 쿼리, 어드민 페이지에 영향**을 미칩니다. 승인합니까?

> 기획자 답변:

**BQ-03: 기존 데이터 처리**

현재 프로덕션에 배포된 사이트에 실사용자 데이터가 있습니까? **Clean slate (DB 초기화) 허용 여부**를 확인해야 합니다. 런칭 전이라면 초기화가 가장 안전합니다.

> 기획자 답변:

### Priority B — 특정 기능 스트림을 차단하는 질문

**BQ-04: 추첨 공정성 로그 법적 요건**

기획서에 "공정성 증명 로그 영구 보관 (추첨 시각, 시드값, 결과 전체)"이 명시되어 있습니다. 이것이 **법적 준수 요건**(예: 경품류 제공에 관한 불공정거래행위의 유형 및 기준)인가요, 아니면 사용자 신뢰를 위한 운영 정책인가요? 법적 요건이라면 변조 방지(append-only + checksum) 수준의 구현이 필요합니다.

> 기획자 답변:

**BQ-05: PASS 본인인증 범위**

기획서에 "본인 인증 (PASS) → 논의 필요"로 표기되어 있습니다. 현재 SMS 인증(Twilio)은 구현되어 있으나, PASS 본인인증은 별도 계약 및 비용이 필요합니다. **V1에서 필수인가요, 아니면 SMS 인증으로 대체 가능한가요?** 1인 1계정 정책의 강도가 이에 따라 달라집니다.

> 기획자 답변:

**BQ-06: V1 운영 주체 확인**

기획서: "V1은 (주)아이콘스가 직접 주최하는 이벤트만 운영." 이 경우:
- 판매자 정보 탭(S-06 ③탭)은 (주)아이콘스 정보를 하드코딩해도 되나요?
- 정산 관리(어드민 ⑥)는 자사 이벤트이므로 V2로 미뤄도 되나요?
- 이벤트 개설 폼에서 "주최자 직접 입력" 필드는 고정값으로 처리해도 되나요?

> 기획자 답변:

**BQ-07: 온라인 팝업 IP 테마 페이지**

기획서: "IP 비주얼 테마(대표색, 배경, 로고) 설정 + 미리보기." 이 페이지의 비주얼 에셋(이미지, 로고)은 누가 제작/제공합니까? **개발자가 만드는 CMS 페이지인가요, 디자이너가 제작하는 커스텀 페이지인가요?** 구현 방식이 완전히 달라집니다:
- CMS 방식: 색상/로고/배경 설정 → 템플릿 기반 자동 생성 (개발 1~2주)

> 기획자 답변:

**BQ-08: 쿠폰 시스템 V1 범위**

## 8. 기술 우려 사항

### 8.1 스키마 마이그레이션 리스크

`performances` → `events` 리네이밍은 **모든 API 엔드포인트, 프론트엔드 쿼리, 어드민 페이지, 테스트 코드**에 영향을 미칩니다. 한 번에 완전히 전환하는 것이 가장 안전하며, 중간 상태(두 모델 공존)는 복잡도만 높입니다. Phase A에서 일괄 처리를 권장합니다.

### 8.2 추첨 공정성 로그

기획서: "공정성 증명 로그 영구 보관 (추첨 시각, 시드값, 결과 전체)." 기술적으로 PostgreSQL에 append-only 테이블 + SHA-256 체인 해싱으로 구현 가능합니다. 단, **법적 요건 수준**에 따라 구현 복잡도가 달라집니다:
- 운영 정책 수준: 단순 로그 테이블 (1~2일)
- 법적 감사 수준: 변조 방지 체인 + 외부 검증 가능 해시 (3~5일)

### 8.3 1계정 3기기 정책

현재 인증은 **Stateless JWT**입니다. 기기 제한을 구현하려면:
- 옵션 A: Valkey에 디바이스 세션 저장 (세션당 TTL, 4번째 기기 접속 시 가장 오래된 세션 무효화)
- 옵션 B: DB에 디바이스 테이블 추가

옵션 A가 성능상 유리하며, 현재 Valkey 인프라를 활용 가능합니다. 단, JWT 무효화 메커니즘(블랙리스트)과 병행해야 합니다.

### 8.4 알림 엔진

기획서는 10+ 트리거 타입, 이메일+SMS 멀티채널, 크로스 타입 중복 방지를 요구합니다. 이것은 사실상 **독립적인 서브시스템**입니다. pg-boss(이미 기술 스택에 포함)를 잡큐로 활용하고, 알림 템플릿/라우팅/중복방지 레이어를 별도 NestJS 모듈로 설계해야 합니다.

### 8.5 온라인 팝업 IP 테마 페이지

이벤트마다 "대표색, 배경, 로고"를 설정하여 커스텀 랜딩을 생성하는 기능입니다. 구현 방식에 따라 복잡도가 크게 다릅니다:
- **CMS 방식** (권장): 색상/이미지 설정값을 DB에 저장, 템플릿 기반 렌더링. 디자인 자유도는 낮지만 운영 비용 최소.
- **커스텀 방식**: HTML/CSS를 직접 업로드. 자유도 높지만 매 이벤트마다 디자이너 작업 필요.

### 8.6 1인 개발 대역폭

기획서의 전체 V1 범위는 약 **5~6.5개월**의 풀타임 개발을 요구합니다. V1 범위를 "최소 런칭 가능 세트"로 좁히면 개발 기간을 3~4개월로 단축할 수 있습니다. **V1 범위 축소**에 대한 기획자의 판단이 필요합니다.
