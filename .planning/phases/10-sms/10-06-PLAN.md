    **Step 5: [Review #7] 429 응답 통일 분석:**
    - `@Throttle` IP 초과 → NestJS ThrottlerGuard → `ThrottlerException` (HTTP 429) → HttpExceptionFilter → `{ statusCode: 429, message: 'ThrottlerException: Too Many Requests' }`
    - SmsService phone 초과 → `BadRequestException({ statusCode: 429, message: '잠시 후 다시 시도해주세요' })` → HttpExceptionFilter → `{ statusCode: 400, error: ..., message: '...' }`

    **문제:** SmsService가 `BadRequestException` 안에 `statusCode: 429`를 넣으면 실제 HTTP 상태는 400이고 body.statusCode만 429. 프론트가 `ApiClientError.statusCode`를 HTTP status에서 가져오면 400으로 인식.

    **해결:** SmsService의 phone axis 429를 `HttpException(body, 429)`로 변경하거나, 프론트 분기가 body.statusCode도 체크하도록. 가장 깔끔한 해결: SmsService에서 phone axis 제한 시 `HttpException` 직접 사용:

    ```typescript
    // sms.service.ts에서 BadRequestException 대신:
    import { HttpException, HttpStatus } from '@nestjs/common';
    throw new HttpException(
      { statusCode: 429, message: '잠시 후 다시 시도해주세요', retryAfterMs },
      HttpStatus.TOO_MANY_REQUESTS,
    );
    ```

    **본 plan에서의 action:** sms.controller.ts 변경과 함께 sms.service.ts의 phone axis `BadRequestException({ statusCode: 429 })` 3곳을 `HttpException(..., 429)`로 교정. 이로써 ThrottlerException(429)과 service phone limit(429) 모두 HTTP 429 반환. 프론트 D-20 매핑 정확.

    주의: `sms.service.ts`는 이미 Plan 05에서 작성된 상태. 3줄만 수정 — `BadRequestException({ statusCode: 429, ... })` → `throw new HttpException({ statusCode: 429, message: '잠시 후 다시 시도해주세요', retryAfterMs: ... }, HttpStatus.TOO_MANY_REQUESTS)`. import에 `HttpException, HttpStatus` 추가.
