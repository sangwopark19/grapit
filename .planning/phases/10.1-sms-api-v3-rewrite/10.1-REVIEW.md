# Phase 10.1: Code Review Report

**Reviewed:** 2026-04-20
**Depth:** standard
**Files Reviewed:** 9 (of 10 requested — `.env.example` was not accessible via Read; see note below)
**Status:** issues_found

## Summary

Phase 10.1 cleanly replaces Infobip 2FA with the v3 `/sms/3/messages` send-only API plus self-managed OTP via Valkey Lua. The v3 request shape (`messages[].sender / destinations[].to / content.text`), leading `+` strip, pre-store-before-send pattern, and atomic 4-way Lua verify are all implemented correctly. The PR #16 carry-over fixes (Issue 1 attempts reset via pipeline, Issue 2 phone-axis counter rollback, Issue 3 groupId=5 REJECTED detection) are all in place with corresponding regression tests.

The biggest actionable gap is **InMemoryRedis drift** — `SmsService` now uses `redis.pipeline()`, `redis.pttl()`, and `redis.decr()`, none of which are implemented in the dev-fallback mock. A developer running without `REDIS_URL` but with Infobip credentials set would crash the send path. This is not a production bug (production hard-fails without `REDIS_URL`), but it breaks dev ergonomics and test-with-real-Infobip flows.

Additional warnings cover the verified-flag short-circuit (phone-verification enumeration vector), secret handling concerns (PII in logs + no sender-ID validation), and a few info-level polish items.

`.env.example` could not be loaded by Read (permission denied — likely project ignore rule) so env-file coverage is not reviewed; the env-variable set is verified indirectly via `sms.service.ts` constructor and test fixtures.

## Warnings

### WR-01: InMemoryRedis mock missing `.pipeline()`, `.decr()`, and `.pttl()` — breaks dev without REDIS_URL

**File:** `apps/api/src/modules/booking/providers/redis.provider.ts:14-248`
**Issue:** `SmsService` in Phase 10.1 now depends on three `ioredis` methods that are not implemented in `InMemoryRedis`:
- `redis.pipeline()` — used at `sms.service.ts:198` for the atomic OTP SET + attempts DEL.
- `redis.decr(key)` — used at `sms.service.ts:226` for phone-axis send counter rollback on 5xx/network errors.
- `redis.pttl(key)` — used at `sms.service.ts:157` for the 429 `retryAfterMs` response.

Additionally, the existing `InMemoryRedis.set(key, value, opts?)` signature is incompatible with the `ioredis` variadic call at `sms.service.ts:155`: `this.redis.set(cooldownKey, '1', 'PX', RESEND_COOLDOWN_MS, 'NX')`. The third positional arg becomes `'PX'` (a string), not an options object — `opts?.nx` is `undefined`, `opts?.ex` is `undefined`, and the NX/TTL semantics are silently lost.

Production hard-fails without `REDIS_URL` (line 268–273), so this is not a prod bug. But any developer who sets `INFOBIP_API_KEY/BASE_URL/SENDER` in local dev without also setting `REDIS_URL` will hit runtime crashes the first time `send-code` is called, and cooldown/TTL tests against the mock are silently broken.

**Fix:** Either (a) extend `InMemoryRedis` to cover the surface used by SMS:

```typescript
// redis.provider.ts — InMemoryRedis additions

// ioredis-variadic SET: set(key, value, 'PX', ms, 'NX') / set(key, value, 'EX', s)
async set(key: string, value: string, ...args: unknown[]): Promise<string | null> {
  // Parse variadic flags
  const flags = args.map((a) => (typeof a === 'string' ? a.toUpperCase() : a));
  const nx = flags.includes('NX');
  const pxIdx = flags.indexOf('PX');
  const exIdx = flags.indexOf('EX');
  const pxMs = pxIdx >= 0 ? Number(flags[pxIdx + 1]) : undefined;
  const exSec = exIdx >= 0 ? Number(flags[exIdx + 1]) : undefined;

  if (nx && this.store.has(key)) return null;
  this.store.set(key, value);

  const ttlMs = pxMs ?? (exSec !== undefined ? exSec * 1000 : undefined);
  if (ttlMs !== undefined) {
    const prev = this.ttls.get(key);
    if (prev) clearTimeout(prev);
    this.expiries.set(key, Date.now() + ttlMs);
    this.ttls.set(key, setTimeout(() => {
      this.store.delete(key);
      this.ttls.delete(key);
      this.expiries.delete(key);
    }, ttlMs));
  }
  return 'OK';
}

async decr(key: string): Promise<number> {
  const next = Number(this.store.get(key) ?? '0') - 1;
  this.store.set(key, String(next));
  return next;
}

async pttl(key: string): Promise<number> {
  if (!this.store.has(key) && !this.sets.has(key)) return -2;
  const expiry = this.expiries.get(key);
  if (!expiry) return -1;
  const remaining = expiry - Date.now();
  return remaining > 0 ? remaining : -2;
}

pipeline(): {
  set: (...a: unknown[]) => ReturnType<InMemoryRedis['pipeline']>;
  del: (...a: string[]) => ReturnType<InMemoryRedis['pipeline']>;
  exec: () => Promise<Array<[Error | null, unknown]>>;
} {
  const ops: Array<() => Promise<[Error | null, unknown]>> = [];
  const self = this;
  const chain = {
    set(...a: unknown[]) {
      ops.push(async () => {
        try { return [null, await self.set(a[0] as string, a[1] as string, ...a.slice(2))]; }
        catch (e) { return [e as Error, null]; }
      });
      return chain;
    },
    del(...keys: string[]) {
      ops.push(async () => {
        try { return [null, await self.del(...keys)]; }
        catch (e) { return [e as Error, null]; }
      });
      return chain;
    },
    async exec() { return Promise.all(ops.map((fn) => fn())); },
  };
  return chain;
}
```

Or (b) explicitly require `REDIS_URL` in dev too (loosen production-only check to "any non-test env"), and remove `InMemoryRedis` entirely. Option (a) keeps the zero-config dev story that the `[redis]` warning advertises.

Also note: the existing `set()` signature needs to go — the variadic form is the live caller contract.

### WR-02: `verifyCode` short-circuits rate limit on `sms:verified` flag → enumeration vector

**File:** `apps/api/src/modules/sms/sms.service.ts:258-262`
**Issue:** When `sms:verified:{e164}` is set (TTL 600s), `verifyCode` returns `{ verified: true }` without ever checking the user-supplied `code` argument and without touching the phone-axis verify counter or OTP store. Two follow-on risks:

1. **Enumeration.** An unauthenticated attacker who POSTs `/sms/verify-code { phone, code: "000000" }` for arbitrary phone numbers can distinguish (within a 10-minute window) which phones just completed SMS verification, regardless of the `code` they send. The response shape differs: verified phones return `{ verified: true }`, unverified return `{ verified: false, message: ... }` or `GoneException`. Signup completion timing is leaked.
2. **Contract violation.** Downstream consumers that call `verifyCode` as a proof-of-possession check — instead of correlating a session token issued at verify-time — would be fooled by any arbitrary code input. The current controller has no session binding between `/send-code` and `/verify-code`, so "this request proves phone ownership" is not really true after the 10-minute flag is set.

**Fix:** Bind the verified flag to a server-issued opaque token and return that token (not a phone-global flag) at verify-time; require the token on consumers (signup, password-reset). Short-term mitigation: move the `alreadyVerified` check *after* the rate-limit increment and return `{ verified: true }` only when the supplied `code` matches a still-stored OTP OR a session token is presented:

```typescript
async verifyCode(phone: string, code: string, sessionToken?: string): Promise<VerifyResult> {
  const e164 = parseE164(phone);
  // ... dev mock block ...

  // D-07 rate limit runs regardless of prior verification — counter is the
  // only signal that resists enumeration on /verify-code.
  const verifyCount = await this.atomicIncr(
    `sms:phone:verify:${e164}`, VERIFY_PHONE_WINDOW_SEC,
  );
  if (verifyCount > VERIFY_PHONE_LIMIT) { /* 429 */ }

  // Idempotent re-verify only when the caller proves session continuity.
  if (sessionToken) {
    const stored = await this.redis.get(`sms:verified:token:${sessionToken}`);
    if (stored === e164) return { verified: true };
  }

  // Otherwise fall through to Lua verify as today.
}
```

If the token redesign is too big for this phase, at minimum add a note in the controller JSDoc that `verifyCode` is NOT a standalone auth primitive and that consumers must correlate with the session that originally sent the OTP.

### WR-03: `INFOBIP_SENDER` accepts any string — invalid sender leaks to runtime as Infobip 400s

**File:** `apps/api/src/modules/sms/sms.service.ts:97`, `apps/api/src/modules/sms/infobip-client.ts:22-27`
**Issue:** `INFOBIP_SENDER` is trimmed and checked only for emptiness. In KR production, an alphanumeric sender ID (e.g. `Grapit`) will be silently rewritten by MNOs — the comment on `infobip-client.ts:21` acknowledges this, but there is no validation that `sender` is a numeric/KISA-registered format. A config typo (`INFOBIP_SENDER=Grapit` instead of `0212345678`) bricks all SMS delivery in production with every request returning `InfobipApiError(400)` AND keeping the user's 5/hour send quota fully consumed (per the "4xx = keep counter" abuse-mitigation rule at `sms.service.ts:220-228`). Real users get permanently rate-limited by a config mistake.

**Fix:** Validate `INFOBIP_SENDER` at construction for KR deployments. Numeric-only with reasonable length is the simplest guard:

```typescript
// sms.service.ts constructor, after extracting `sender`
if (isProduction && sender && !/^[0-9]{4,15}$/.test(sender)) {
  throw new Error(
    `[sms] INFOBIP_SENDER must be a KISA-registered numeric ID (got "${sender.length <= 3 ? sender : sender.slice(0, 2) + '***'}"). ` +
      `Alphanumeric senders are rewritten by KR MNOs and cause silent delivery failure.`,
  );
}
```

If Grapit intends to support non-KR senders later, relax the check to "numeric OR ≤11 alphanumeric chars" per Infobip sender-ID docs. Also consider surfacing the *first* 4xx-groupId=5 rejection with a WARN log that explicitly calls out sender misconfiguration as the most likely cause.

### IN-01: Phone numbers logged as PII — consider masking

**File:** `apps/api/src/modules/sms/sms.service.ts:116,149,158,170-172,210,241,252,269-271,294,297,302,318`
**Issue:** Every log line includes `phone: e164` in full. Under PIPA / GDPR, full mobile numbers are personal data and should be masked at the application layer unless there's a documented need. Sentry breadcrumbs inherit these tags too.
**Fix:** Add a helper `maskE164('+821012345678') → '+8210****5678'` and use it in every `logger.*` call. Keep the full number only in the Valkey keys (where TTL bounds retention).

### IN-02: `parseE164` error throws `Error` (not `BadRequestException`) — leaks as 500

**File:** `apps/api/src/modules/sms/phone.util.ts:46`
**Issue:** `parseE164` throws a plain `Error('올바른 휴대폰 번호를 입력해주세요')` on `ParseError`. `sendVerificationCode` / `verifyCode` call `parseE164` at the top without a try/catch. The controller's `ZodValidationPipe` already rejects malformed input, so this is unlikely to trigger in practice — but a phone that passes the regex (`+[1-9]\d{6,14}`) yet fails `libphonenumber-js` full parse (e.g. `+99999999999` — invalid country code) would surface as HTTP 500 rather than 400.
**Fix:** Throw a `BadRequestException` directly, or wrap `parseE164` at the service layer:

```typescript
// phone.util.ts
import { BadRequestException } from '@nestjs/common';
// ...
throw new BadRequestException('올바른 휴대폰 번호를 입력해주세요');
```

Alternatively, add a lightweight ParseError → BadRequestException translation at the service boundary to keep `phone.util.ts` framework-agnostic.

### IN-03: `redis.decr` rollback can go negative on crash windows

**File:** `apps/api/src/modules/sms/sms.service.ts:225-227`
**Issue:** The rollback `DECR sms:phone:send:{e164}` on 5xx can race: if process A increments to 3, crashes before DECR, and process B increments to 4, DECR from A (if retried later) would put the counter at 3 — not strictly incorrect. But in "Infobip sendSms succeeds but logger throws" pathological cases, DECR could put the counter at 0/-1, temporarily granting the user an extra slot. Impact is low (next hour the key TTL resets) but the bound isn't enforced.
**Fix:** Use a Lua script that floors at 0: `if redis.call('GET', KEYS[1]) and tonumber(redis.call('GET', KEYS[1])) > 0 then return redis.call('DECR', KEYS[1]) else return 0 end`. Low priority.

### IN-05: Integration test uses invalid KR E.164 phone numbers

**File:** `apps/api/test/sms-throttle.integration.spec.ts:120-122, 139-145, 164-167, 178-186`
**Issue:** Test phone strings like `+82010012345678` (13 digits after +82, mobile numbers are 10-11 digits) and `+82010000099` (10 digits with a leading `0` that E.164 strips) are not valid Korean mobile E.164.
