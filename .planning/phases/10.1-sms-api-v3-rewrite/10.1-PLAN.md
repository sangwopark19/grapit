# Phase 10.1: SMS API v3 전환 — Plan Index

## Goal (from ROADMAP.md)

Infobip `/2fa/2/pin` 레거시 경로에서 `/sms/3/messages` v3로 SMS 발송을 전환하여 `INFOBIP_APPLICATION_ID` / `INFOBIP_MESSAGE_ID` 의존을 제거하고, OTP 생성·검증을 Valkey 기반 자체 구현으로 단순화한다.

## Success Criteria

1. `InfobipClient`가 `POST /sms/3/messages` 호출로 재작성되어 `applicationId`/`messageId` 필요 없이 API Key + Base URL만으로 동작
2. 6자리 숫자 OTP 생성 + Valkey `sms:otp:{e164}` 저장 (TTL 180s) + verify 매칭이 자체 구현됨
3. 환경변수는 `INFOBIP_API_KEY` + `INFOBIP_BASE_URL` + `INFOBIP_SENDER` 3개로 축소. `APPLICATION_ID`/`MESSAGE_ID` 제거
4. 기존 Phase 10의 rate limiting(IP + phone axis Lua counter) + 30s resend cooldown + mock 모드 분기 모두 유지
5. `pnpm --filter @grapit/api test`가 0 실패로 통과 (Phase 10 테스트 마이그레이션 완료)

## Implementation Flow (Interface First)

1. **Wave 1 Plan 01**: env 계약 고정 (`INFOBIP_API_KEY`/`BASE_URL`/`SENDER`) + 운영 문서 선반영. 운영자가 "더 이상 2FA Application 만들지 말 것" 지침을 즉시 받는다.
2. **Wave 1 Plan 02**: 테스트가 구현 계약 선언. `sendSms`/`SendSmsResult`/`VERIFY_AND_INCREMENT_LUA` 결과 타입 및 `sms:otp:{e164}` 키 스키마가 spec 안에 명시됨.
3. **Wave 2 Plan 03**: `infobip-client.ts` GREEN 전환 — spec의 v3 body shape, 4xx/5xx 에러, AbortSignal.timeout 5초 어서션 충족.
4. **Wave 3 Plan 04**: `sms.service.ts` GREEN 전환 — env 3종, `crypto.randomInt`, Valkey 5 key 스키마, Lua 4분기 결과 분기 구현. auth.service.ts 호출부 무영향(응답 shape 불변).
5. **Wave 4 Plan 05**: controller 단위 + testcontainers integration에서 회귀 없음 확인. 신규 `VERIFY_AND_INCREMENT_LUA`가 실 Valkey EVAL에서 작동하는 smoke 추가.
6. **Wave 5 Plan 06**: 자동 게이트 통과 → 실 SMS 1회 smoke → 운영 문서 최종 재검토.

## Key Decisions (from 10.1-CONTEXT.md)

| ID | Decision | Rationale |
|----|----------|-----------|
| LOCKED | `/sms/3/messages` v3 endpoint | 2024-12 Infobip 출시, applicationId 제거됨 |
| LOCKED | 6자리 숫자 OTP + `crypto.randomInt` | OWASP A02 — Math.random 예측 가능성 제거 |
| LOCKED | Valkey `sms:otp:{e164}` TTL 180s 직접 저장 | PIN mapping 제거, 해시 불필요 (Valkey = trust zone) |
| LOCKED | GET + 비교 + DEL을 Lua 원자 실행 + `sms:verified:{e164}` TTL 600s 저장 | race condition 방지, 회원가입 직후 비번 설정 단계 재확인 용 |
| LOCKED | env 3종 체계 (API_KEY/BASE_URL/SENDER) | 2FA Application/Message Template 관리 부담 제거 |
| LOCKED | 본문 `[Grapit] 인증번호 ${pin} (3분 이내 입력)` | Phase 10 Message Template과 동일, i18n은 Phase 10.2+ |
| Discretion | Lua로 verify + attempts 원자화 | Phase 10 `ATOMIC_INCR_LUA` 패턴과 일관성 유지 |
| Discretion | Sender ID 형식 검증 scope 밖 | Infobip portal 수준에서 통제 |

## Threat Model Summary (STRIDE)

집계된 주요 위협:

| Category | Primary Mitigation |
|----------|---------------------|
| **Tampering** (OTP brute-force) | Lua atomic verify + 5회 제한 + phone/IP axis 중첩 rate limit |
| **Spoofing** (예측 가능 OTP) | `node:crypto.randomInt` CSPRNG |
| **Information Disclosure** (OTP/key 유출) | OTP는 로그/Sentry 미노출, API Key는 constructor 경유 메모리만 |
| **Denial of Service** (Infobip 지연) | AbortSignal.timeout 5초 + cooldown rollback (5xx/timeout) |
| **Repudiation** (4→3 env 전환 이력) | 문서 메타 노트 + ROADMAP 업데이트 + SUMMARY 기록 |
