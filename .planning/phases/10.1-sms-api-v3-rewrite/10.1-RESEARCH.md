## 1. Infobip SMS API v3 구조

### 1.1 Endpoint

```
POST https://{baseUrl}/sms/3/messages
```

- `baseUrl`: 계정별 고유 서브도메인 (예: `xxxxx.api.infobip.com`)
- 인증: `Authorization: App {apiKey}` 헤더 (Phase 10과 동일)
- Content-Type: `application/json`

### 1.2 Request Body Schema (최소 필드)

```json
{
  "messages": [
    {
      "from": "SenderID",
      "destinations": [
        { "to": "821012345678" }
      ],
      "text": "Your message here"
    }
  ]
}
```

**필수 필드:**
- `messages[].from` — 발신자 ID (alphanumeric 3-11자 또는 numeric 3-16자)
- `messages[].destinations[].to` — 수신 번호 (E.164에서 `+` 제거)
- `messages[].text` — 메시지 본문

**Phase 10의 `/2fa/2/pin`과의 차이:**
- ❌ `applicationId` 불필요
- ❌ `messageId` 불필요
- ❌ Message Template 사전 생성 불필요
- ❌ 2FA Application 사전 생성 불필요
- ✅ PIN 생성·저장·검증은 우리가 직접 구현

### 1.3 Response Schema

```json
{
  "messages": [
    {
      "messageId": "2250be2d4219-3af1-78856-aabe-1362af1edfd2",
      "status": {
        "groupId": 1,
        "groupName": "PENDING",
        "id": 26,
        "name": "MESSAGE_ACCEPTED",
        "description": "Message sent to next instance"
      },
      "destination": "821012345678"
    }
  ],
  "bulkId": "2034072219640523072"
}
```

- `status.groupId`: 1=PENDING, 2=UNDELIVERABLE, 3=DELIVERED, 4=EXPIRED, 5=REJECTED
- MVP 기준: groupId=1 받으면 "발송 성공" 처리. 실제 delivery 확인은 webhook 필요 (deferred).

### 1.4 에러 응답

```json
{
  "requestError": {
    "serviceException": {
      "messageId": "BAD_REQUEST",
      "text": "to: must be a valid phone number"
    }
  }
}
```

**HTTP status codes:**
- 200: 발송 큐잉 성공
- 400: 입력 오류 (수신 번호 형식, 본문 누락 등)
- 401: API Key 인증 실패
- 403: Sender ID 권한 없음 / KISA 미등록 (한국 특정)
- 429: Infobip 전역 rate limit (계정 plan에 따라 다름)
- 5xx: 일시적 오류 — 재시도 권장

## 2. 구현 패턴

### 2.1 InfobipClient 최소 API

```typescript
// apps/api/src/modules/sms/infobip-client.ts (rewrite target)

export class InfobipApiError extends Error {
  constructor(
    public status: number,
    public infobipCode: string | null,
    message: string,
  ) {
    super(message);
  }
}

export interface SendSmsResult {
  messageId: string;
  status: string;      // e.g. "MESSAGE_ACCEPTED"
  groupId: number;     // 1=PENDING, etc.
}

export class InfobipClient {
  constructor(
    private readonly baseUrl: string,
    private readonly apiKey: string,
    private readonly sender: string,
  ) {}

  async sendSms(e164: string, text: string): Promise<SendSmsResult> {
    const to = e164.replace(/^\+/, '');
    const url = `${this.baseUrl}/sms/3/messages`;

    const res = await fetch(url, {
      method: 'POST',
      headers: {
        Authorization: `App ${this.apiKey}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        messages: [
          { from: this.sender, destinations: [{ to }], text }
        ],
      }),
      signal: AbortSignal.timeout(5000),
    });

    if (!res.ok) {
      const body = await res.text().catch(() => '');
      throw new InfobipApiError(
        res.status,
        extractInfobipCode(body),
        `Infobip API ${res.status}: ${body.slice(0, 200)}`,
      );
    }

    const data = await res.json();
    const msg = data.messages?.[0];
    if (!msg) throw new InfobipApiError(500, null, 'Infobip response missing messages[0]');

    return {
      messageId: msg.messageId,
      status: msg.status?.name ?? 'UNKNOWN',
      groupId: msg.status?.groupId ?? 0,
    };
  }
}
```

### 2.2 OTP 생성 & 저장

```typescript
// crypto.randomInt는 Math.random보다 예측 불가능 (보안)
import { randomInt } from 'node:crypto';

function generateOtp(): string {
  return String(randomInt(100000, 1000000));  // 100000~999999 (6자리)
}

// Valkey 저장
const otp = generateOtp();
await redis.set(`sms:otp:${e164}`, otp, 'PX', 180_000);  // TTL 180s

// 메시지 발송
await client.sendSms(e164, `[Grapit] 인증번호 ${otp} (3분 이내 입력)`);
```

### 2.3 Verify 로직 (Lua atomic)

```lua
-- verify-and-increment.lua
-- KEYS[1] = sms:otp:{e164}
-- KEYS[2] = sms:attempts:{e164}
-- KEYS[3] = sms:verified:{e164}
-- ARGV[1] = user-provided code
-- ARGV[2] = max attempts (5)
-- ARGV[3] = verified flag TTL seconds (600)

local stored = redis.call('GET', KEYS[1])
if stored == false then
  return {'EXPIRED', 0}
end

local attempts = tonumber(redis.call('INCR', KEYS[2]))
redis.call('EXPIRE', KEYS[2], 900)  -- 15min window

if attempts > tonumber(ARGV[2]) then
  redis.call('DEL', KEYS[1], KEYS[2])
  return {'NO_MORE_ATTEMPTS', 0}
end

if stored == ARGV[1] then
  redis.call('DEL', KEYS[1], KEYS[2])
  redis.call('SETEX', KEYS[3], tonumber(ARGV[3]), '1')
  return {'VERIFIED', attempts}
end

return {'WRONG', ARGV[2] - attempts}  -- remaining attempts
```

이로써 "OTP 확인 + 시도 횟수 증가 + 성공 플래그 설정"이 원자적으로 처리됨. race condition 없음.

## 4. 알려진 함정 (Pitfalls)

### 4.1 Sender ID 형식 — Infobip은 검증하지 않음
- `INFOBIP_SENDER` 값이 KISA 등록 안 된 번호여도 Infobip API는 200 반환
- 실제 도달은 한국 통신사 단계에서 차단됨 (Delivery Report에서만 확인 가능)
- MVP: 대응 안 함. Sender 등록 책임은 운영자에게.

### 4.2 E.164 `+` 제거
- Phase 10 `infobip-client.ts:50`에서 이미 `e164.replace(/^\+/, '')` 패턴 사용
- Phase 10.1에서도 동일하게 처리

### 4.3 `crypto.randomInt` vs `Math.random`
- `Math.random`은 예측 가능 — OTP 생성에 부적합 (OWASP A02:2021 Cryptographic Failures)
- `node:crypto.randomInt(100000, 1000000)`은 CSPRNG 기반

### 4.4 Lua script — EVAL vs EVALSHA
- Phase 10이 `redis.eval(SCRIPT, ...)` 사용 (cache 없음)
- Phase 10.1도 동일 패턴 유지 (script 캐싱 최적화는 Phase 12 이후)

### 4.5 AbortSignal.timeout Node 22 지원 확인
- Node 22.22.x LTS에서 `AbortSignal.timeout(ms)` 정상 지원 (Phase 10 이미 검증됨)
- 테스트에서 fake timer와 충돌 가능 — Phase 10 이미 대응 완료

### 4.6 테스트 mocking — fetch vs undici
- Phase 10 `infobip-client.spec.ts`가 `global.fetch` mock 사용
- Phase 10.1도 동일 패턴. undici 직접 import 금지.

## 5. 검증 기준 (Validation Architecture)

### 5.1 Unit 테스트 범위
- InfobipClient.sendSms: 성공/4xx/5xx/timeout 4종
- SmsService.sendVerificationCode: OTP 생성 → Valkey 저장 → sendSms 호출 확인
- SmsService.verifyCode: 정답/오답/만료/5회 초과/verified 플래그 저장

### 5.2 Integration 테스트 (testcontainers)
- Phase 10의 `sms-throttle.integration.spec.ts` 재활용
- OTP 생성·저장·검증 flow end-to-end

### 5.3 E2E 테스트
- Phase 10의 `signup-sms.spec.ts` 재활용 (API shape 동일하므로 무수정)

### 5.4 Manual smoke (Phase 10.1 final human verification)
- 로컬 `pnpm dev` + `.env` 3종 INFOBIP env 주입
- 회원가입 flow에서 본인 번호로 SMS 수신
- 수신된 6자리 입력 → verified
- 동일 번호로 30초 이내 재발송 시도 → cooldown 메시지
- 틀린 코드 5번 입력 → "재발송 필요" 메시지
