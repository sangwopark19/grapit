### WR-01: InMemoryRedis mock missing `.pipeline()`, `.decr()`, `.pttl()` — breaks dev without REDIS_URL

**Files modified:** `apps/api/src/modules/booking/providers/redis.provider.ts`, `apps/api/src/modules/booking/providers/__tests__/redis.provider.spec.ts`
**Commit:** `cf14d03`
**Applied fix:** Extended `InMemoryRedis` with ioredis-parity surface:
- `set(key, value, ...args)` now supports BOTH the legacy options-object form (`{ nx, ex }` — used internally by `evalLockSeat`/`evalVerifyAndIncrement`) AND the ioredis variadic form (`'PX', ms, 'NX'` / `'EX', s`) used by `SmsService.sendVerificationCode` and `CacheService.set`.
- Added `decr(key)` for phone-axis send counter rollback.
- Added `pttl(key)` returning milliseconds with ioredis semantics (-2 missing, -1 no TTL).
- Added `pipeline()` returning a chainable builder with `set`/`del`/`exec`. `exec()` resolves to `Array<[Error|null, unknown]>` tuples, matching ioredis pipeline result shape so `SmsService` can iterate `results.some(([opErr]) => opErr)` unchanged.

Added 5 new tests in `redis.provider.spec.ts` covering variadic SET + NX + PX/EX, `decr`, `pttl` edge cases (-2/-1/positive), and pipeline SET+DEL round-trip. Backward compat verified — existing BookingService integration tests continue to pass.

### WR-02: `verifyCode` short-circuits rate limit on `sms:verified` flag → enumeration vector

**Files modified:** `apps/api/src/modules/sms/sms.service.ts`, `apps/api/src/modules/sms/sms.controller.ts`, `apps/api/src/modules/sms/sms.service.spec.ts`
**Commit:** `5c263ee`
**Applied fix:** Short-term mitigation per review (session-token redesign deferred as TODO). Reordered `verifyCode`:

1. Phone-axis verify counter (D-07 Lua atomic INCR+EXPIRE) now runs FIRST, before any idempotent-verify short-circuit.
2. `sms:verified:{e164}` GET check moved to AFTER the counter — still idempotent, but enumeration probes count toward the 10/900s limit.
3. Added JSDoc on both `SmsService.verifyCode` and `SmsController.verifyCode` explicitly warning that `{ verified: true }` is NOT a standalone auth primitive and consumers must correlate with the original `/send-code` session. `TODO(sms-session-token)` marker added for the longer-term token redesign.

Added 2 regression tests: (a) counter increments BEFORE the GET short-circuit even when flag is set, (b) 11th probe still throws 429 even with flag set (no unbounded short-circuit). Both tests assert `mockRedis.eval.mock.invocationCallOrder[0] < mockRedis.get.mock.invocationCallOrder[0]`.

### WR-03: `INFOBIP_SENDER` accepts any string — invalid sender leaks to runtime as Infobip 400s

**Files modified:** `apps/api/src/modules/sms/sms.service.ts`, `apps/api/src/modules/sms/sms.service.spec.ts`
**Commit:** `2167e12`
**Applied fix:** Added constructor-time validation for `INFOBIP_SENDER` when `NODE_ENV=production`:
- Regex `/^[0-9]{4,15}$/` enforces KISA-registered numeric format (4-15 digits).
- Rejects alphanumeric senders (e.g. `Grapit`) that KR MNOs silently rewrite and cause 4xx cascades; our existing 4xx rollback policy keeps the phone-axis counter, so a sender-ID typo would permanently drain every user's 5/hour quota with zero delivery.
- Error message masks the sender value (`SE***` instead of full) to avoid log leakage.
- Non-production envs skip the check for dev-friendliness (matches existing dev-mock pattern).

Added 5 regression tests: alphanumeric reject, short-digit reject, valid 10-digit pass, masking assertion, non-production bypass.
