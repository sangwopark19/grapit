    **B. §1.3 재작성 — "2FA Application 생성" → "발신자(Sender ID) 확정":**
    기존 §1.3 본문 전체를 다음 구조로 교체:
    - 개요: `/sms/3/messages`는 Application 개념이 없다. 요청 시 `from` 필드에 직접 sender를 넣으므로 **한 번 포털에서 sender를 확정해두면 끝**.

    **C. §1.4 재작성 — "Message Template 생성" → "메시지 본문 규격":**
    기존 §1.4 본문 전체를 다음으로 교체:
    - `/sms/3/messages`는 Template 개념이 없다. 요청 body의 `messages[].text`에 본문을 직접 넣는다.
    - Grapit 고정 본문: `[Grapit] 인증번호 <OTP> (3분 이내 입력)` (OTP는 `crypto.randomInt` 6자리 숫자)
    - EUC-KR 1 SMS = 140 bytes (한글 약 45자). 본문 길이 확인: 한글 14자 + 숫자 6자 + 기호 → 1 SMS 안에 수용.
    - 한국 통신사 자동 태그 `[국제발신]`이 앞에 붙으므로, 본문에 `[Grapit]` 브랜드 고지는 유지.
    - i18n(+66, +1 등)용 영문 본문은 Phase 10.2 이후 검토 (deferred).

    **D. §1.5 갱신 — API Key Scope 최소권한:**
    - 기존 "2fa:pin:send / 2fa:pin:manage / 2fa:manage" 스코프 표를 다음으로 교체:
      ```
      | Scope ID         | 용도                                 | 운영 키 필수 |
      |------------------|--------------------------------------|--------------|
      | sms:send         | POST /sms/3/messages 호출            | 필수         |
      | sms:receive      | DLR webhook(delivery receipt) 수신  | 선택 (deferred) |
      | numbers:manage   | Sender ID 관리                      | 선택 (KISA 등록 시만) |
      ```
    - 운영 키 권장: `sms:send`만. `numbers:manage`는 KISA 등록/변경 시점에만 분리된 bootstrap 키로 부여 후 즉시 폐기.
    - §1.5.5 인증 헤더는 `Authorization: App {INFOBIP_API_KEY}` 형식 그대로 유지 (v3도 동일).

    **E. §1.6 Free Trial 검증 cURL 갱신:**
    기존 `POST /2fa/2/pin` + `POST /2fa/2/pin/{pinId}/verify` 2단계 cURL 블록 전체를 삭제하고 하기 v3 단일 호출 예시로 대체:
    ```bash
    curl -X POST "https://{baseUrl}/sms/3/messages" \
    ```
