<domain>
## Phase Boundary

Phase 10이 Infobip **2FA PIN API (`POST /2fa/2/pin`)** 로 구현됐으나, Infobip이 2024-12 SDK v5.0.0에서 **일반 SMS API v3 (`POST /sms/3/messages`)**로 API를 단순화하면서 `applicationId`/`messageId` 의존 없이 API Key + Base URL만으로 SMS 발송이 가능해짐. Grapit의 회원가입 OTP 인증은 "단순 SMS 발송 + 6자리 코드 검증"에 해당하므로 2FA PIN API의 Application/Message Template 관리가 과도함.

**이 phase의 범위:**
1. `apps/api/src/modules/sms/infobip-client.ts`를 `/sms/3/messages` 호출로 재작성
2. `apps/api/src/modules/sms/sms.service.ts`에 PIN 생성 + Valkey 저장 + verify 매칭 로직 자체 구현
3. 환경변수에서 `INFOBIP_APPLICATION_ID`, `INFOBIP_MESSAGE_ID` 제거. `INFOBIP_SENDER` 추가
4. 기존 테스트(sms.service.spec, infobip-client.spec 등) 마이그레이션
5. DEPLOY-CHECKLIST.md 재작성 — Application/Message Template 생성 섹션 제거
6. INFOBIP-SETUP-GUIDE.md 업데이트 (§1.3, §1.4 완전 제거/교체)

**범위 밖:**
- KISA 발신번호 등록 절차 (별도 트랙, 가이드만 유지)
- 국제 번호 대응 로직 (Phase 10에서 이미 구현됨)
- Rate limiting, cooldown, mock 모드 (Phase 10 코드 재사용)
- Frontend `phone-verification.tsx` (API 호출 경로 동일, 변경 불필요)

</domain>

<decisions>
## Implementation Decisions

### SMS API 선택
- **LOCKED: `POST /sms/3/messages` v3 endpoint** (2024-12-23 출시, applicationId 제거됨)
- 레거시 `/2fa/2/pin` 경로 완전 제거
- Request body 형식:
  ```json
  {
    "messages": [{
      "from": "<INFOBIP_SENDER>",
      "destinations": [{ "to": "<E164 without +>" }],
      "text": "[Grapit] 인증번호 <OTP> (3분 이내 입력)"
    }]
  }
  ```

### OTP 생성 로직
- **LOCKED: 6자리 숫자, crypto.randomInt(100000, 1000000)** (보안상 Math.random 금지)
- 저장: Valkey key `sms:otp:{e164}`, TTL 180s (Phase 10의 `PIN_MAPPING_TTL_MS` 유지)
- 저장 값: 6자리 OTP 문자열 (해시 불필요 — Valkey 접근 자체가 신뢰 경계)

### Verify 로직
- **LOCKED: Valkey GET 후 단순 string 비교 + DEL** (idempotent verify 지원 안 함 — Phase 10의 `allowMultiplePinVerifications` 동작 재현 불가)
- 대안: verify 성공 시 별도 `sms:verified:{e164}` 플래그 키를 TTL 600s로 저장 (회원가입 직후 비번 설정 단계에서 재확인 가능)
- attempts remaining 카운터: `sms:attempts:{e164}` Valkey key, 5회 초과 시 PIN 삭제 + GoneException

### 환경변수 축소
- **제거**: `INFOBIP_APPLICATION_ID`, `INFOBIP_MESSAGE_ID`
- **유지**: `INFOBIP_API_KEY`, `INFOBIP_BASE_URL`
- **추가**: `INFOBIP_SENDER` (KISA 등록 발신번호 또는 Free Trial용 alphanumeric. 프로덕션은 KISA 등록 필수)
- dev mock 분기 조건: `INFOBIP_API_KEY || INFOBIP_BASE_URL || INFOBIP_SENDER` 중 하나라도 missing이면 `isDevMock=true`

### 메시지 본문
- **LOCKED**: `[Grapit] 인증번호 ${pin} (3분 이내 입력)` (Phase 10 Message Template과 동일)
- 국제 번호의 경우 i18n 처리는 Phase 10.2 이후로 deferred

### 테스트 전략
- Phase 10 `infobip-client.spec.ts` 전면 재작성 (sendSms/verifyPin → sendSms only)
- Phase 10 `sms.service.spec.ts` OTP 생성·저장·검증 테스트 추가
- Phase 10 fixtures (`infobip-send-response.json`, `infobip-verify-response.json`) 재작성 또는 제거
- E2E `signup-sms.spec.ts`는 그대로 (API 응답 shape 동일)

### Claude's Discretion
- Lua script로 verify + attempt increment 원자화할지 여부 (현재 판단: Phase 10 패턴과 일관성 위해 Lua 도입 권장)
- Sender ID 형식 검증 (alphanumeric vs numeric vs KISA 등록 번호) — `INFOBIP_SENDER` env 값 그대로 통과시킴
- Toss secret 유출 감지 같은 추가 보안 검증 — scope 밖
</decisions>

<specifics>
## Specific Ideas

- **InfobipClient 최소 구현**: `sendSms(e164: string, text: string): Promise<{ messageId: string; status: string }>` 하나의 public method만. verify는 제거 (자체 구현).
- **SmsService 변경 포인트**:
  - `sendVerificationCode(phone)`: 6자리 OTP 생성 → Valkey 저장 → InfobipClient.sendSms(e164, `[Grapit] 인증번호 ${otp} (3분 이내 입력)`)
  - `verifyCode(phone, code)`: Valkey GET `sms:otp:{e164}` → 일치 여부 → attempts counter 증가 또는 verify 성공 플래그 저장
- **mock 모드**: `isDevMock=true`일 때 sendSms 호출 skip, logger.warn 출력. verifyCode는 `000000`만 통과 (Phase 10 패턴 유지).
- **에러 매핑**:
  - Infobip 4xx (발송 실패): `sms.service.ts`에서 `BadRequestException` + cooldown 유지
  - Infobip 5xx/network: cooldown rollback (Phase 10 `sms.service.ts:138-143` 패턴)
  - Valkey eval 실패: retry 없음, 그냥 500 전파
- **Lua script 재사용**: Phase 10의 `ATOMIC_INCR_LUA`는 IP/phone axis rate limiting용이라 그대로 유지. OTP verify용으로 별도 `VERIFY_AND_INCREMENT_LUA` 추가 고려.

</specifics>

<deferred>
## Deferred Ideas

- **국제 번호 i18n**: 한국어 외 메시지 템플릿은 별도 phase (10.2?) 혹은 Phase 12 UX 현대화와 통합
- **allowMultiplePinVerifications 완전 대응**: 현재는 verify 성공 후 `sms:verified:{e164}` 플래그만 TTL 600s로 보관. 다중 verify는 사용 사례 발생 시 재검토
- **KISA 발신번호 자동 검증**: sender ID 형식 체크 (예: `01[0-9]{8,9}` regex) — 운영 경험 쌓인 후 판단
- **Infobip webhook (delivery receipt)**: 발송 성공/실패를 비동기로 받는 경로. 현재 /sms/3/messages 응답만으로 충분하므로 보류
</deferred>
