<decisions>
## Implementation Decisions

### 확정된 사실 (locked — 계획 수립 시 전제)

- **API 측 정상:** CI 의 "Smoke test login" step 에서 `curl` 로 동일 endpoint 호출 시 HTTP 200 + accessToken + `Set-Cookie: refreshToken` 을 정상 반환. API 기동과 endpoint 자체는 정상.
- **DB 측 정상:** seed 가 `admin@grapit.test` / `TestAdmin2026!` 를 정상 삽입. `pw_len=97` (argon2 hash 정상 길이) — "Verify seed" step 로그 확인.
- **실패 지점 특정:** Playwright 의 helper 가 `POST /api/v1/auth/login` 을 보낼 때 응답 body 가 `{"statusCode":401,"message":"Unauthorized",...}` — NestJS validateUser 의 custom 메시지 `"이메일 또는 비밀번호가 일치하지 않습니다"` 가 **아니므로** passport-local 이 `req.body.email` / `req.body.password` 자체를 읽지 못한 것. (validateUser 가 호출되지 않았다는 뜻)
- **진단 우선 원칙:** 해결 전 서버 측에 도달한 실제 `req.body` / `req.headers` / `req.rawBody` 를 관찰해야 한다. 서버가 무엇을 받았는지 확인하기 전의 수정은 모두 추측이다.

### 시도했지만 실패한 우회 (repeat 하지 말 것 — 모두 동일 401)

1. `page.request.post('/api/v1/auth/login', { data: {...}, headers: {...} })` via Next.js rewrites (proxy)
2. `page.request.post('http://localhost:8080/api/v1/auth/login', ...)` API 직접 호출
3. `data: JSON.stringify({...})` 명시 직렬화
4. `Accept: application/json` 추가
5. `request.newContext()` 로 browser context 격리

### 남은 가설 (검증 대상)

- Playwright 의 Chromium 기반 request layer 가 body 를 streaming / multipart 로 encode → NestJS express body-parser 파싱 실패
- `Content-Length` 누락 또는 `Transfer-Encoding: chunked` 로 body 전송 → passport-local 이 `req.body` 를 못 읽음
- NestJS 의 global middleware / ValidationPipe 가 특정 헤더 조합에서 body 를 drop

### 제안 진단 방법 (순서대로 권장)

1. **[FIRST]** NestJS `AuthController.login` 에 임시 debug middleware 를 주입하여 `req.body`, `req.headers`, `req.rawBody` 를 구조적으로 로그. CI 에서 curl 요청 과 Playwright helper 요청 **양쪽** 을 동일 middleware 로 관찰. 이 로그 **없이 수정하지 않는다**.
2. **[IF helper-side issue]** Playwright 내부에서 Node native `fetch()` (undici) 로 login 직접 호출 시도 — Chromium 기반 request 를 완전히 우회하는 helper 대체 검증.
3. **[LAST RESORT]** Wireshark / tcpdump 로 wire-level 요청 비교 (로컬 CI 재현이 가능할 때만).

### Claude's Discretion

- 진단 middleware 의 정확한 주입 위치 (main.ts vs AuthModule vs AuthController decorator) 및 제거 시점 판단
- helper 수정의 구체적 API 선택 (node fetch 대체 vs APIRequestContext 옵션 조정 vs 서버 측 body-parser 설정 변경)
- CI 로그 파싱/비교 방식 (행별 diff vs JSON structured)
- 3개 fixme 복구 후 회귀 방지를 위한 CI smoke 강화 여부

</decisions>
