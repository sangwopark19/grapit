### content-length (결정적 증거)

| Source | content-length | Body actual content (reconstructed) | Server response |
|--------|---------------|-------------------------------------|-----------------|
| curl smoke (hardcoded) | 57 | `{"email":"admin@grapit.test","password":"TestAdmin2026!"}` | HTTP 200 + accessToken |
| Playwright probe (env vars) | **26** | `{"email":"","password":""}` | HTTP 401 Unauthorized |

26 bytes 는 정확히 `{"email":"","password":""}` 의 길이. content-type 은 양쪽 모두 `application/json`, transfer-encoding chunked 도 양쪽 모두 absent. **Playwright 의 body 가 단순히 빈 문자열을 담고 있다.**

### Root cause chain (재구성)

1. `gh secret list` → `TEST_USER_EMAIL`, `TEST_USER_PASSWORD` 가 repo secret 에 **존재하지 않음** (GCP/Toss/R2 secret 만 존재).
2. ci.yml line 161-162 의 `${{ secrets.TEST_USER_EMAIL }}` 는 secret 미존재 시 **빈 문자열** 을 env value 로 주입 (GitHub Actions 표준 동작).
3. apps/web/e2e/helpers/auth.ts:32-33 와 apps/web/e2e/diagnostic/auth-login-probe.spec.ts:5-6 모두 `process.env['TEST_USER_EMAIL'] ?? 'admin@grapit.test'` 사용.
4. JS nullish coalescing (`??`) 는 `null` 또는 `undefined` 만 catch — 빈 문자열은 falsy 지만 nullish 가 아님 → 통과.
5. email/password 가 빈 문자열로 evaluated → Playwright 가 `{"email":"","password":""}` (26 bytes) 전송.
6. NestJS body-parser 가 정상 파싱 → passport-local LocalStrategy 가 빈 credentials 를 reject → 401.

curl smoke 가 HTTP 200 인 이유: ci.yml line 145 의 curl 명령은 **하드코딩된 리터럴** 사용 (env var 미사용).

## Why diff.md 의 bodyType/bodyKeys/rawBody 가 모두 null 인가

진단 미들웨어가 `app.use(createAuthReqDebugMiddleware())` 로 등록되어 NestJS 내부 body-parser **앞** 에서 실행되었기 때문이다. NestJS 의 body-parser 는 internally 등록되어 user `app.use()` 호출 시점에는 아직 chain 에 없다. 결과적으로 req.body 와 req.rawBody 모두 undefined/null 상태에서 미들웨어가 실행됨.

이는 **Plan 01 의 설계 결함이지만 진단 결과에는 영향 없음** — content-length 헤더만으로 root cause 를 확정할 수 있었다. Plan 05 Wave 4 에서 어차피 모두 제거되므로 수정하지 않음.
