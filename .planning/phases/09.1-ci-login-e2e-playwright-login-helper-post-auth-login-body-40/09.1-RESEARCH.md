1. `page.request.post('/api/v1/auth/login', ...)` via Next.js rewrites (proxy)

4. `Accept: application/json` 추가
5. `request.newContext()` 로 browser context 격리

- Playwright Chromium 기반 request layer 가 body 를 streaming / multipart 로 encode → NestJS express body-parser 파싱 실패

## Summary

Phase 09.1 은 CI 한정으로만 재현되는 **transport-layer 미스매치** 를 규명하여 `test.fixme()` 처리된 3건을 복구하는 urgent tech-debt phase 다.

**핵심 관찰 (3가지 독립 신호가 일관됨):**

1. CI 의 curl (동일 네트워크·동일 endpoint·동일 credentials) → HTTP 200. API·DB·seed·네트워크 경로는 모두 정상.
2. Playwright helper 응답이 `{"statusCode":401,"message":"Unauthorized", ...}` 포맷 — 이는 **`HttpExceptionFilter` (apps/api/src/common/filters/http-exception.filter.ts:22-30)** 가 통과시킨 `UnauthorizedException` 이며, message 가 generic `"Unauthorized"` 라는 것은 `AuthService.validateUser` (auth.service.ts:130-145) 내부의 custom throw 가 **아니라** passport-local 의 기본 `Unauthorized` fallback 임을 의미. 즉 **validate 가 호출조차 되지 않음**.
3. passport-local 의 validate 가 호출되지 않는 유일한 시나리오는 `req.body[usernameField]` (여기선 `email`) 또는 `req.body[passwordField]` (여기선 `password`) 가 undefined 인 경우 — 즉 **body-parser 가 body 를 파싱하지 못했거나, body 자체가 도달하지 않았다** [CITED: github.com/nestjs/docs.nestjs.com/issues/875; stackoverflow "AuthGuard('local') validate never called"].

**Primary recommendation:** **진단 middleware 를 먼저 주입하여 서버 측에 도달한 raw request 를 관찰**. 이 증거 없이 수정하지 않는다. 증거 기반으로 원인 범주 (A/B/C) 를 결정하고 그에 대응하는 최소 수정을 적용한다 (자세한 결정 트리는 아래 `## Architecture Patterns` §Decision Tree 참조).

**Key enabling facts:**

- Playwright `APIRequestContext.post({data: <object>})` 는 object → JSON.stringify 후 `Content-Type: application/json` 자동 세팅. 다만 **`Content-Length` 는 공식 문서에 명시되어 있지 않고 Chromium network stack 이 결정** — 상황에 따라 `Transfer-Encoding: chunked` 로 전송될 가능성 존재 [CITED: playwright.dev APIRequestContext docs].
- Express body-parser 는 `Content-Type: application/json` 매칭 시 chunked body 도 정상 파싱 (streaming 기반) — 다만 **헤더 조합이 비정형일 때 드물게 drop** 하는 리포트 존재 [CITED: expressjs/body-parser#201].
- Node.js 22 LTS (CI 고정) 는 native `fetch` (undici 기반) 를 제공 — Playwright 의 Chromium request 를 완전히 우회하는 **현실적인 대체 경로**.
- **Next.js 16 dev mode rewrites 의 POST body drop 이슈** 가 문서화되어 있다 [CITED: vercel/next.js#50200, #87071] — 그러나 현재 helper 는 이미 `:8080` 직접 호출로 rewrites 를 우회하므로 이 원인은 **사실상 배제** (우회 실패 로그가 locked fact 에 있음).

### Pattern 1: Diagnostic Middleware (일회성, 제거 대상)

**What:** Playwright 와 curl 양쪽 요청을 서버 측에서 **동일 파이프로 관찰** 하여 `req.body`, `req.headers`, `req.rawBody` 를 구조적으로 로그.

**When to use:** 진단 단계 (Task 1). 원인 확정 후 즉시 제거.

**Why first:** 5가지 client-side 우회가 모두 실패한 시점에서, 서버가 무엇을 받았는지 확인하지 않은 수정은 **모두 추측**. 이 phase 의 가장 중요한 단일 행동은 raw 증거 확보.

**Example (권장 패턴):**

```typescript
// Source: docs.nestjs.com/middleware + docs.nestjs.com/faq/raw-body
// Location: apps/api/src/main.ts (bootstrap(), bodyParser 등록 직후)
// CRITICAL: body-parser 뒤에 실행되어야 req.body 가 파싱된 상태로 관찰됨.
//           NestFactory.create 는 기본으로 body-parser 를 먼저 등록하므로,
//           app.use(...) 를 이 지점에 넣으면 순서는 자동으로 맞는다.

if (process.env['DEBUG_AUTH_REQ'] === '1') {
  app.use((req: Request, _res: Response, next: NextFunction) => {
    if (req.method === 'POST' && req.url.includes('/auth/login')) {
      // 구조적 로그 — CI stdout 에서 grep/jq 하기 쉽게 한 줄 JSON.
      console.log(JSON.stringify({
        tag: 'AUTH_LOGIN_DEBUG',
        method: req.method,
        url: req.url,
        headers: req.headers,
        // body 는 이미 body-parser 가 파싱한 상태. undefined 면 "파싱 실패"
        // 의 명확한 증거.
        body: req.body,
        bodyKeys: req.body ? Object.keys(req.body) : null,
        // rawBody 는 기본 제공되지 않음 — 아래 Pattern 2 참조.
        rawBody: (req as { rawBody?: Buffer }).rawBody?.toString('utf8') ?? null,
      }));
    }
    next();
  });
}
```

**Key:** 환경변수 `DEBUG_AUTH_REQ=1` 로 게이트 → CI workflow 에서만 켜고 제거 시 단일 환경변수/블록 삭제로 정리.

### Pattern 2: rawBody 캡처 (필요 시)

**What:** body-parser 가 parse 하기 전의 **원본 바이트** 를 보존하여 `Content-Length` / chunked 인코딩 여부를 검증.

**When to use:** Pattern 1 의 `req.body` 관찰 결과 `{}` 또는 `undefined` 이고, 헤더만으로는 원인이 불분명할 때.

**Why:** Playwright 의 Chromium network stack 이 body 를 실제로 전송했는지 여부를 서버 관점에서 확정.

**Example:**

```typescript
// Source: docs.nestjs.com/faq/raw-body
// Location: apps/api/src/main.ts, NestFactory.create 옵션
const app = await NestFactory.create(AppModule, {
  rawBody: true,  // req.rawBody: Buffer 를 enable
});

// Nest 11 이상에서 공식 지원. body-parser 동작 변경 없음 —
// parse 된 req.body 와 raw Buffer 를 동시에 제공한다.
```

**Note:** 이 옵션은 기동 시점에 설정하므로, Pattern 1 과 달리 **bootstrap 단계에서 결정**. 진단 완료 후 단순히 플래그 제거.

### Pattern 3: Node native fetch helper (최후의 수단)

**What:** Playwright APIRequestContext 를 완전히 우회하고 Node 22 의 native `fetch` (undici 기반) 로 로그인 요청을 전송. Set-Cookie 를 수동 파싱 후 `page.context().addCookies()` 로 browser context 에 replay.

**When to use:** 진단 결과 원인이 **Playwright Chromium network stack 쪽** 으로 확정된 경우 (Decision Tree §A).

**Why:** 현재 helper 는 이미 `request.newContext()` 로 격리했지만 여전히 Chromium 기반 request 파이프를 쓴다. node native fetch 는 undici 라는 완전히 다른 HTTP client 를 사용 → 인코딩 차이로 인한 원인이면 즉시 해소.

**Example:**

```typescript
// Source: nodejs.org/learn/getting-started/fetch (Node 22 native fetch)
//         MDN Set-Cookie header
// Location: apps/web/e2e/helpers/auth.ts (replace loginAsTestUser body)
import type { Page } from '@playwright/test';

export async function loginAsTestUser(page: Page): Promise<void> {
  const email = process.env['TEST_USER_EMAIL'] ?? 'admin@grapit.test';
  const password = process.env['TEST_USER_PASSWORD'] ?? 'TestAdmin2026!';
  const apiURL = process.env['E2E_API_URL'] ?? 'http://localhost:8080';

  // undici 기반 fetch — Playwright Chromium stack 완전 우회.
  const res = await fetch(`${apiURL}/api/v1/auth/login`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ email, password }),
  });

  if (!res.ok) {
    const text = await res.text().catch(() => '<unreadable>');
    throw new Error(`[loginAsTestUser] Login failed: ${res.status} ${text}`);
  }

  // Set-Cookie 파싱 — Node fetch 는 cookie jar 를 안 다루므로 수동.
  // Node 22 에서는 res.headers.getSetCookie() 가 있다.
  const setCookieValues = res.headers.getSetCookie();  // string[]
  const refreshCookie = setCookieValues
    .map((raw) => raw.split(';')[0])  // "refreshToken=xxxxxx"
    .find((kv) => kv.startsWith('refreshToken='));

  if (!refreshCookie) {
    throw new Error('[loginAsTestUser] refreshToken cookie missing in Set-Cookie');
  }

  const [, value] = refreshCookie.split('=');
  await page.context().addCookies([{
    name: 'refreshToken',
    value,
    domain: 'localhost',
    path: '/',
    httpOnly: true,
    secure: false,       // CI 는 http — secure=true 였다면 cookie 적용 안 됨
    sameSite: 'Lax',
  }]);
}
```

**Caveat:** `secure: false` 로 replay 해야 http://localhost 에서 유효. 서버의 `setRefreshTokenCookie` (auth.controller.ts:254-262) 는 `secure: true` + `sameSite: 'none'` 으로 설정하지만, Playwright context 에서는 **test 환경에 맞게 override** 해야 한다 (이 점은 현재 helper 의 `addCookies` 에도 동일 이슈 있음 — STATE 의 "프로덕션 소셜로그인 쿠키 SameSite 버그 수정" 선례 참고).

### Decision Tree (진단 후 원인 분기)

```
[Task 1] 진단 middleware 주입 + CI 재현
        ↓
    req.body = ?
    ┌───────┴───────────┐
    │                   │
  {} or undefined    {email, password}
    │                   │
    │                   ↓
    │              (원인 C) passport-local 내부 문제
    │               → @nestjs/passport 버전 조사, strategy 재등록
    │               → 드문 케이스, plan 에서 fallback 으로만
    │
    ├── req.headers['content-type']?
    │   ├── "application/json" 이면 → (원인 A)
    │   └── 다른 값 또는 누락  → (원인 B)
    │
    ↓
  (원인 A) Playwright body 가 application/json 인데 파싱 실패
    → Transfer-Encoding / Content-Length 확인
    → rawBody 관찰 (Pattern 2) → bytes 가 오면 body-parser 설정 조정
    → bytes 가 안 오면 → node fetch 우회 (Pattern 3)
  (원인 B) Content-Type 이 잘못 또는 누락
    → Playwright headers 옵션 반영 여부 확인
    → Pattern 3 로 전환 (node fetch 는 명시적 헤더 세팅)
```

**중요:** 이 결정트리는 **관찰 결과에 따라 갈라진다**. planner 는 이 트리를 task 로 펼치되, 각 분기에 대해 조건부 task 를 사전에 정의하고 실제 진단 결과로 1개만 실행한다.

### Anti-Patterns to Avoid

- **증거 없이 수정:** 5차례 시도해 학습된 교훈. 진단 middleware 없이 6번째 client-side 추측 수정 시도 **금지**.
- **서버 측 body-parser 옵션 변경을 1차 선택으로:** production 에 영향을 주는 변경 (`@nestjs/common` 의 `json()` 제거, limit 조정 등) 은 회귀 위험. **helper 수정이 production 접점 0 이므로 항상 우선**.
- **진단 middleware 상주:** 진단용 `console.log` 가 커밋에 남으면 프로덕션 로그 오염 + 유저 credential 유출 리스크. **제거는 해결 commit 과 같은 PR 에서 동시에**.
- **"test.fixme 만 풀고 root cause 확정 전 green 호도":** passport-local 동작에 대한 오해 (validateUser 호출 안 됨 ≠ credentials 불일치) 가 재발 가능.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| HTTP body 파싱 (JSON) | 커스텀 JSON.parse middleware | NestJS 기본 body-parser | 이미 main.ts 에서 자동 등록. edge case (chunked, large body, malformed UTF-8) 모두 처리됨. [CITED: docs.nestjs.com/faq/raw-body] |
| passport-local field 매칭 | custom Guard 에서 `req.body.email` 수동 추출 | `usernameField: 'email'` 을 strategy 에 설정 (이미 완료) | 이미 local.strategy.ts:10 에 설정됨. field 매칭 자체는 정상. |
| CI 에서 HTTP client 로 login | curl 을 Playwright test 내부에서 shell 실행 | Node native fetch (undici) | child_process 스폰 오버헤드 + ESM/shell quoting 리스크. fetch 가 더 깔끔. |
| Cookie jar 관리 | 수동 Cookie 헤더 조립 | `page.context().addCookies()` (현 helper 방식 유지) | Playwright 의 browser context 가 이후 모든 요청에 자동 첨부. |
| HTTP 요청 diff | 수동 text diff in reviewer head | `diff curl-baseline.txt playwright-req.txt` (structured JSON) | 인간 비교 오류 방지. CI artifact 로 첨부. |

**Key insight:** 이 phase 의 수정은 "새 코드 작성" 이 아니라 **"어느 레이어에 문제가 있는지 관찰로 좁히고, 그 레이어의 공식 대체 메커니즘으로 교체"** 하는 것. 대부분 코드량은 진단 middleware (삭제 예정) + helper 의 fetch 전환이다.

## Common Pitfalls

### Pitfall 1: "validateUser 호출 안 됨 = credentials 불일치" 로 잘못 해석

**What goes wrong:** Passport 문서 경험 부족으로 `401 Unauthorized` 를 무조건 "비밀번호 틀림" 으로 해석, seed/hash/DB 를 반복 확인.

**Why:** NestJS Passport 에서 401 의 출처가 2개 — (a) strategy.validate 내부의 throw, (b) strategy 가 **field 를 못 찾아 호출조차 안 될 때의 기본 fallback**. 메시지 차이 (`"이메일 또는 비밀번호가 일치하지 않습니다"` vs `"Unauthorized"`) 가 유일한 구분선. [CITED: github.com/nestjs/docs.nestjs.com/issues/875]

**How to avoid:** 응답 body 의 `message` 필드를 **항상 먼저 확인**. `"Unauthorized"` (generic) 이면 `req.body` 수신 단계부터 의심. 이미 CONTEXT 의 locked fact 로 확정되었으므로 본 phase 는 이 해석을 전제로 진행.

**Warning signs:** HTTP status 만 보고 수정 시도. credential/DB/seed 반복 확인에 시간 소모.

### Pitfall 2: "local 에서 되니까 CI 특유 네트워크 문제" 로 점프

**What goes wrong:** 로컬 재현 실패 → 즉시 CI 네트워크 (proxy, DNS, firewall) 로 점프하여 불필요한 infra 변경.

**Why:** 로컬 dev 는 **`nest start` (ts-node)** 기반이고 CI 는 **`nest build` → `node dist/main.js`** 기반 [VERIFIED: .github/workflows/ci.yml:95-96, 111]. 컴파일 결과가 동일해야 하지만 source map / emitDecoratorMetadata / body-parser 등록 순서에 미묘한 차이 가능. 하지만 **본 phase 의 ci.yml 는 curl 로 동일 endpoint 를 통과시킴** → 네트워크·빌드 자체는 정상, Playwright client 쪽이 원인일 확률이 더 높음.

**How to avoid:** curl 이 green 인 사실을 **CI 네트워크·서버 빌드 정상 증거** 로 인정하고, client side 에 집중.

**Warning signs:** GitHub Actions runner IP / DNS / Docker network 에 대한 가설만 늘어나고 검증 불가능한 문자열 (timeout, route table) 조사로 시간 소모.

### Pitfall 3: 진단 middleware 의 상주화

**What goes wrong:** `console.log(req.headers)` 가 `ship it` 되어 프로덕션 로그에 credentials (특히 password) 누출.

**Why:** 로그 구조화가 안 되어 있으면 stdout 에 raw header + body 가 흘러가고, Cloud Run logs 는 영구 보존.

**How to avoid:** 진단 middleware 는 **환경변수 gate (`DEBUG_AUTH_REQ=1`)** 로 기본 OFF. 해결 commit 과 같은 PR 에서 제거. 제거 task 를 plan 에 명시적으로 포함.

**Warning signs:** middleware 제거 task 가 plan 에 없음. "나중에 제거하자" 는 주석. `.env.example` 에 `DEBUG_AUTH_REQ` 노출.

### Pitfall 4: Cookie secure/sameSite 의 test vs production 차이 망각

**What goes wrong:** Set-Cookie 가 `Secure; SameSite=None` 으로 내려오는데 Playwright 는 `http://localhost` 기반이라 secure cookie 는 저장·전송되지 않음 → 후속 요청에서 refreshToken 없음.

**Why:** server 측 `auth.controller.ts:254-262` 의 `setRefreshTokenCookie` 는 `httpOnly: true, secure: true, sameSite: 'none'` 고정. CI 의 Playwright 는 http 로 서빙된 API 에 요청 → Chromium 이 Secure cookie 저장 거부.

**How to avoid:** 현재 helper 가 이미 `page.context().addCookies()` 로 강제 주입하므로 이 부분은 우회됨. 다만 **node fetch 대체 구현 시 같은 주입 경로를 유지** 해야 동일 동작. (Pattern 3 예제 참조 — `secure: false` 로 replay)

**Warning signs:** 로그인 응답은 200 인데 이후 navigation 에서 `refreshToken` 이 없다는 에러.

### Pitfall 5: Next.js 16 dev mode rewrites 재도입

**What goes wrong:** planner 가 "`/api/*` 경유로도 되게 하자" 는 nice-to-have 를 추가하다가 Next 16 의 POST body drop 이슈 [CITED: vercel/next.js#50200] 에 다시 걸림.

**Why:** 현재 helper 는 `:8080` 직접 호출로 rewrites 를 우회했고 CONTEXT 의 "시도했지만 실패한 우회 #1" 에 이미 기록됨.

**How to avoid:** helper 경로는 **항상 `apiURL` 환경변수 기반 직접 호출** 로 유지. Next rewrites 를 다시 거치지 않음.

**Warning signs:** PLAN 에 "/api/ 경로로 통일" 류 task 가 등장.

## Code Examples

### §1 진단 middleware (main.ts 패치, 환경변수 gate)

```typescript
// Source: docs.nestjs.com/middleware + docs.nestjs.com/faq/raw-body
// Location: apps/api/src/main.ts, bootstrap() 내부 — app.use(cookieParser()) 이후
import type { NextFunction, Request, Response } from 'express';

// NestFactory.create({ rawBody: true }) 로 먼저 enable.
// 이렇게 하면 req.rawBody 가 Buffer 로 노출된다.
// https://docs.nestjs.com/faq/raw-body

if (process.env['DEBUG_AUTH_REQ'] === '1') {
  app.use((req: Request & { rawBody?: Buffer }, _res: Response, next: NextFunction) => {
    const isAuthLogin = req.method === 'POST' && req.url.includes('/auth/login');
    if (isAuthLogin) {
      // CI stdout 에 한 줄 JSON 으로 — `grep AUTH_LOGIN_DEBUG | jq .` 로 비교.
      console.log(JSON.stringify({
        tag: 'AUTH_LOGIN_DEBUG',
        method: req.method,
        url: req.url,
        headers: req.headers,  // content-type, content-length, transfer-encoding 관찰
        bodyType: typeof req.body,
        bodyKeys: req.body && typeof req.body === 'object' ? Object.keys(req.body) : null,
        bodyEmail: (req.body as { email?: string })?.email ?? null,
        bodyPasswordLen: (req.body as { password?: string })?.password?.length ?? null,
        rawBodyLen: req.rawBody?.length ?? null,
        rawBodyUtf8: req.rawBody?.toString('utf8') ?? null,
      }));
    }
    next();
  });
}
```

### §2 Passport-local field resolution (확인용 — 수정 아님)

```typescript
// Source: passport-local source + docs.nestjs.com/recipes/passport
// File: apps/api/src/modules/auth/strategies/local.strategy.ts (기존 코드)
//
// super() 의 옵션이 req.body 에서 field 를 어떻게 뽑는지 결정:
//   - usernameField: 'email'   → req.body.email 을 읽음
//   - passwordField: 'password' → req.body.password 를 읽음
//
// 둘 중 하나라도 undefined 이면 passport-local 은 validate(...) 를 호출하지
// 않고 바로 fail()  → AuthGuard 가 UnauthorizedException('Unauthorized') 를
// 생성 → HttpExceptionFilter 가 { statusCode: 401, message: 'Unauthorized' }
// 로 응답. 이것이 현재 CI 에서 관찰되는 증상과 정확히 일치.
//
// 수정: 없음. 이 strategy 는 정상. req.body 가 도달하는지가 문제.
@Injectable()
export class LocalStrategy extends PassportStrategy(Strategy, 'local') {
  constructor(private readonly authService: AuthService) {
    super({
      usernameField: 'email',
      passwordField: 'password',
    });
  }

  async validate(email: string, password: string): Promise<ValidatedUser> {
    // 이 메서드가 호출된다는 것은 req.body.email + req.body.password 가
    // 이미 정상적으로 전달됐다는 뜻. 현재 CI 에서는 여기 도달하지 않는다.
    return this.authService.validateUser(email, password);
  }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `page.request.post(...)` (browser-context shared) | `playwright.request.newContext()` (isolated cookie jar) | Playwright 1.19+ | 현재 helper 가 이미 채택. 이 자체는 bug 원인 아님. [CITED: playwright.dev/docs/api/class-apirequestcontext] |
| require() / CommonJS middleware | ES modules only | CLAUDE.md 전역 규칙 | 이 phase 의 모든 코드는 `import`. |
| 커스텀 HTTP client (axios 등) | Node 22 native `fetch` | Node 18+ stable, 20+ 안정화 | 추가 dep 없이 undici 사용. [CITED: nodejs.org/learn/getting-started/fetch] |
| `req.rawBody` 를 커스텀 middleware 로 수집 | `NestFactory.create(App, { rawBody: true })` | NestJS 11 공식 옵션 | 한 줄로 enable. [CITED: docs.nestjs.com/faq/raw-body] |

**Deprecated/outdated:**
- **`httpCredentials` 기본 send behavior:** Playwright 1.45+ 부터는 `httpCredentials.send: 'always'` 를 명시 가능 — 이 phase 에서는 basic auth 를 쓰지 않으므로 무관하지만, 미래 auth-scheme 교체 시 참고. [CITED: github.com/microsoft/playwright issue #30534]

## Open Questions

1. **CI local runner 의 Node 버전이 정확히 Node 22 몇 인가?**
   - What we know: `actions/setup-node@v6` + `node-version: 22` [VERIFIED]
   - What's unclear: 22.x 중 22.22.x 인지, undici subversion 이 무엇인지 — Playwright test worker 프로세스는 이 Node 를 쓴다.
   - Recommendation: 진단 단계에서 `node -v` 를 CI 로그에 명시 출력 (1행 step 추가). local 개발자 machine 도 확인해 diff.

2. **API 서버는 `node dist/main.js` 로 기동. 개발자 local 은 어떤 스크립트?**
   - What we know: CI = `node dist/main.js` [VERIFIED: ci.yml:111]
   - What's unclear: 개발자 local 이 `nest start --watch` 인지 `node dist/main.js` 인지. 차이가 있다면 body-parser 등록 순서 subtle diff 가능.
   - Recommendation: local 재현 시 `pnpm --filter @grapit/api build && node dist/main.js` 로 기동하여 CI 동일 조건 만든 뒤 Playwright 돌려보기.

3. **passport-local 의 내부 버전 (`@nestjs/passport` 11.x, `passport-local` x.y.z) 에 known issue 있는가?**
   - What we know: CLAUDE.md 스택에 `@nestjs/passport 11.0.x` 고정.
   - What's unclear: `passport-local` 자체 버전 및 최근 변경.
   - Recommendation: Fallback 경로로만 남김 (A5 가 관찰 결과 null 차이일 때 활성화).

### Sampling Rate

- **Per task commit:** 해당 task 가 touch 한 layer 만
  - 진단 middleware 수정 → API 재기동 + curl 1회 (`grep AUTH_LOGIN_DEBUG api.log | head -1`)
  - helper 수정 → `pnpm --filter @grapit/web exec playwright test toss-payment.spec.ts --project chromium -x` (failing on first)
  - fixme 해제 → 동일 playwright 명령으로 3건 green 확인
- **Per wave merge:** E2E 전체 (`pnpm --filter @grapit/web test:e2e`) + API unit (`pnpm --filter @grapit/api test`)
- **Phase gate:** CI full suite green (12/12 E2E + 모든 unit) + artifact 에 `AUTH_LOGIN_DEBUG` 로그가 **없음** 확인 (즉 middleware 제거 완료 증명) + 수정 PR diff 가 production 접점 (main.ts 영구 변경, auth.controller 변경) 을 포함하지 않음 확인.

### Layer-Level Assertions (Nyquist proof-of-fix, NOT 그냥 "테스트 통과")

**독립 신호 최소 3개가 일치해야 원인 확정 및 수정 완료로 간주한다.** 한 개만 green 이면 symptom masking 가능성.

| # | Layer | Signal | How Measured | Why Independent |
|---|-------|--------|--------------|-----------------|
| S1 | **서버 측 req 관찰** | `AUTH_LOGIN_DEBUG` 로그에서 `bodyEmail === "admin@grapit.test"` AND `bodyPasswordLen === 17` (TestAdmin2026! 길이) | CI api.log grep | 서버가 실제로 받은 값 — helper 코드와 완전히 독립 |
| S2 | **HTTP wire 관찰** | `AUTH_LOGIN_DEBUG` 의 `headers['content-type']` 에 `application/json` 포함 AND (`headers['content-length']` 존재 OR `rawBodyLen === rawBodyUtf8.length`) | CI api.log grep | transport encoding 의 정상성 — application layer 와 독립 |
| S3 | **response body 검증** | Playwright helper 가 받은 응답 status === 200 AND body 에 `accessToken` 포함 AND Set-Cookie 헤더에 `refreshToken=` 포함 | helper 수정본 내부의 throw 로직 또는 명시적 assert | 서버 round-trip 결과 — 서버 로그와 독립 |
| S4 | **E2E 복구 증거** | `toss-payment.spec.ts` 3건이 `test.fixme` 없이 green (CI 12/12) | playwright html reporter / CI exit code | 실제 user-facing flow — 단위/진단과 독립 |
| S5 | **curl baseline 보존** | 기존 CI "Smoke test login" step 이 HTTP 200 유지 | CI step exit code 0 | 수정이 서버 regression 을 일으키지 않았다는 증거 |

**Proof-of-fix 요건:**

- S1 + S2 + S3 + S4 + S5 가 모두 green 이면 수정 완료.
- S4 만 green 이고 S1/S2 가 설명 없이 green 이면 → **원인이 fuzzy 하게 해소된 것** (symptom masking 의심) → planner 는 이 경우 "원인 범주 문서화" task 를 추가로 요구.

**최소 독립 신호 수:** **3** (S1, S3, S4). S2, S5 는 회귀 방지용 보강 증거. S1 이 절대 조건 — S1 없이 green 이 되면 **다음 환경 변화 (Playwright/Node/NestJS 업그레이드) 시 재발 가능성 높음**.
