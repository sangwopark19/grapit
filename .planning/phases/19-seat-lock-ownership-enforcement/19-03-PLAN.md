<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> reservation API | Untrusted orderId, amount, and seat list enter prepare/confirm. |
| reservation API -> BookingService/Valkey | API validates object-level authorization against Valkey. |
| reservation API -> Toss | External payment confirmation must not run unless pre-check passes. |
| Toss -> DB sold transition | Post-confirm state transition requires atomic lock consume. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-01 | Tampering / Elevation of privilege | `prepareReservation`, `confirmAndCreateReservation` | mitigate | Tasks 1 and 2 call `assertOwnedSeatLocks` before pending creation/return and before Toss confirm. |
| T-19-02 | Tampering | confirm cleanup | mitigate | Task 2 removes broad `unlockAllSeats(userId, reservation.showtimeId)` from confirm success path and relies on targeted consume. |
| T-19-03 | Tampering / DoS | Valkey Lua | mitigate | This plan consumes the cluster-compatible helpers from Plan 19-02; no raw Valkey script is added here. |
| T-19-04 | Elevation of privilege | existing pending `orderId` | mitigate | Task 1 re-checks stored pending reservation seats before returning existing `orderId`. |
| T-19-05 | Tampering / Repudiation | post-Toss sold transition | mitigate | Task 2 consumes locks before DB transaction and runs Toss cancel compensation if consume fails. |
| T-19-06 | Repudiation | frontend false success | transfer | Plan 19-04 handles visible failed UI; this plan returns `409 Conflict` messages through Nest exceptions. |
</threat_model>
