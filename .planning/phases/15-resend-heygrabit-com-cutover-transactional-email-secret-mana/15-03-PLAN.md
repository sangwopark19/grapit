## Plan 03 Design Decisions (REVIEWS MEDIUM M5 — `:latest` vs explicit pin)

- **Decision:** `--update-secrets RESEND_FROM_EMAIL=resend-from-email:latest` 의 `:latest` 표기 유지 (Task 2).
- **Rationale:** `.github/workflows/deploy.yml` L121 이 이미 `:latest` 를 사용하고 있어 (REVIEWS M5 — "keep consistency — do not introduce drift"), Plan 03 에서 명시 version pin (`resend-from-email:<N>`) 을 쓰면 다음 CI/CD 자동 배포가 `:latest` 로 다시 resolve 되어 state drift 가 발생한다. Cloud Run revision 은 revision start 시 `:latest` 를 실제 version 번호로 resolve 하고 revision lifetime 동안 고정하므로 (RESEARCH §Pattern 2), runbook 에서 `:latest` 를 쓰더라도 실행 시점에 Task 1 에서 추가한 신규 version 으로 resolve 됨.
- **Auditability compensation:** `:latest` 사용의 auditability 약점은 15-HUMAN-UAT.md Wave 3 Audit Log 에 (a) Task 1 의 신규 version 번호, (b) Task 1 의 이전 version 번호 (rollback pin — `--filter='state=ENABLED' --sort-by='~createTime' --limit=1` 으로 명시 조회, REVIEWS M3), (c) Task 2 의 신규 revision ID 를 기록하여 보완한다.
- **If user later decides to pin explicitly:** deploy.yml 도 동일하게 pin 하도록 같이 수정해야 한다 (drift 방지). 이는 본 phase 범위 밖.
</decisions>
