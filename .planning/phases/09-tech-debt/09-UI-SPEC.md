# Phase 9 — UI Design Contract

> Visual and interaction contract for 기술부채 청산. Generated by gsd-ui-researcher.

> **Phase nature:** Backend/infrastructure-dominant. UI impact limited to three surfaces:
> (1) 약관 Dialog MD 본문 렌더링 + 법률 검토 배너 (DEBT-02),
> (2) admin-booking-detail-modal의 `paidAt` null empty state 표시 + 포맷 일관성 (DEBT-04),
> (3) 좌석맵 locked 좌석 클릭 시 parent toast 안내 문구/변형 확정 (DEBT-03).
>
> DEBT-01 (Resend email), DEBT-05 (Toss E2E), DEBT-06 (useShowtimes dead code 제거)은 사용자 가시 UI 영향 없음 — 이메일 본문은 React Email 시스템 이메일로 분리되어 UI-SPEC 범위 외.

### MD 약관 본문 타이포그래피 (DEBT-02 신규 명세)

MD 파일로 분리되는 `terms-of-service.md`, `privacy-policy.md`, `marketing-consent.md` 본문은 Dialog 안에서 다음 규격으로 렌더링한다. (Tailwind `@tailwindcss/typography` 플러그인 사용 금지 — 번들 증가 회피. 대신 prose-like 유틸 직접 선언.)

| 마크다운 요소 | 클래스 | 세부 |
|----------|-------|------|
| `<h1>` (문서 제목) | 숨김 처리 (`sr-only`) | 이미 `DialogTitle`이 제목 역할. 중복 방지 |
| `<h2>` (조항 제목) | `text-base font-semibold text-gray-900 mt-6 first:mt-0` | 16px, weight 600, 위 24px 간격 |
| `<h3>` (소항목) | `text-caption font-semibold text-gray-800 mt-4` | 14px, weight 600, 위 16px 간격 |
| `<p>` (본문) | `text-caption leading-relaxed text-gray-700 mt-2` | 14px, line-height ~1.625, 위 8px 간격 |
| `<ul>` / `<ol>` | `text-caption leading-relaxed text-gray-700 mt-2 ml-5 list-disc` (ol은 `list-decimal`) | 들여쓰기 20px |
| `<li>` | 기본 (`mt-1`) | 리스트 항목 간 4px |
| `<strong>` | `font-semibold text-gray-900` | 굵게 + 더 진한 색 |
| `<a>` | `text-primary underline hover:text-primary/80` | #6C3CE0 underline |
| `<hr>` | `my-4 border-gray-200` | 구분선 16px 상하 여백 |

구현 방식: MD → HTML 변환은 빌드타임에 `?raw` import + 런타임 단순 markdown-to-JSX (예: `react-markdown` 또는 수동 파서). `react-markdown` 채택 시 `components` prop으로 위 클래스 매핑 적용. 라이브러리 선정은 planner 재량.

Source: 기존 `apps/web/app/globals.css` typography tokens 재활용. prose-alike 매핑은 본 UI-SPEC에서 최초 정의.

### 법률 검토 배너 (DEBT-02)

약관 MD 본문 최상단에 "법률 검토 전 초안" 배너 표시 (D-07 기준 "배너 형태로 잠시 유지"):

| 속성 | 값 |
|------|-----|
| 배경 | `bg-warning-surface` (#FFFBEB) |
| 보더 | `border border-warning/30` (좌측 `border-l-4 border-l-warning` 강조) |
| 텍스트 색 | `text-[#8B6306]` (admin-booking-table의 PENDING_PAYMENT와 동일한 황갈색 — warning foreground) |
| 아이콘 | `AlertTriangle` (lucide, 16px, `text-warning`) 좌측 정렬 |
| 패딩 | `px-4 py-3` (16px 좌우, 12px 상하) |
| 라운딩 | `rounded-md` (6px) |
| 간격 | 아이콘-텍스트 gap 8px, 배너 하단 24px (`mb-6`) |
| 문구 | "본 약관은 런칭 전 법률 검토를 거쳐 교체될 초안입니다." |
| 노출 조건 | 3개 문서 **모두**에 동일 배너 노출. 런칭 후 법률 검토 완료 시 단일 boolean flag로 전체 숨김 (`const IS_DRAFT = true`) |

배너는 Phase 9에서 **신규** — 기존 코드베이스에 warning 배너 패턴 없음. 단, 색 토큰 (`--color-warning`, `--color-warning-surface`)은 `globals.css:43-44`에 이미 정의.

Source: `apps/web/app/globals.css` lines 5-46. 배너 설계는 본 UI-SPEC에서 최초 정의.

### DEBT-03: 좌석맵 locked 좌석 toast

현재 `booking-page.tsx:193,251`에서 이미 사용 중인 문구 — Phase 9에서 **확정 + 변형 통일**.

| Element | Copy | 변형 |
|---------|------|------|
| Locked 좌석 클릭 toast | "이미 다른 사용자가 선택한 좌석입니다" | `toast.info` (기존 인라인 style → Toaster `classNames.info`로 일원화 권장) |
| 409 Conflict 후 취소 toast | "이미 다른 사용자가 선택한 좌석입니다" | `toast.info` (locked와 동일 문구로 통일) |
| 최대 선택 초과 toast | "최대 4석까지 선택할 수 있습니다. 다른 좌석을 먼저 해제해주세요." | `toast.error` (기존 유지) |
| Lock API 일반 실패 toast | "일시적인 오류가 발생했습니다. 잠시 후 다시 시도해주세요." | `toast.error` (기존 유지) |

**Toast 변형 결정 근거:**
- `info` (보라 `--color-info: #6C3CE0` / `--color-info-surface: #F3EFFF`): 사용자 잘못이 아닌 동시성 상황 — locked 좌석. 경고가 아닌 안내.
- `warning` 미사용: Grapit은 warning을 "법률 검토 배너" 같은 서비스 제공자 쪽 알림에 reserve. 좌석 선점은 타 사용자에 의한 정상 상태.
- `error` (빨강): 사용자 입력/시스템 오류에만 사용.

**구현 노트 (executor 참조):**
- 기존 `booking-page.tsx:193-195, 251-253`의 인라인 `style={{ backgroundColor, color }}` 방식은 Toaster `classNames.info`에 `bg-info-surface text-info border-info/20`을 추가하여 제거. sonner.tsx에 `info` 클래스 추가 권장.
- 테스트 회귀 수정 (DEBT-03)은 production 코드 vs 테스트 중 어느 쪽을 고칠지 planner 단계에서 확인 (D-19). 기대 동작은 **locked 좌석 클릭 시 `onSeatClick` 호출 + parent가 toast** — 이 contract는 변경 불가.

## Interaction States

### DEBT-02: 약관 Dialog 오픈 흐름

```
[회원가입 step 2] → 약관 목록 표시 (기존)
  ↓ 사용자가 "보기" 클릭
[Dialog Open] → DialogContent 렌더 (max-h-[80vh], overflow-y-auto, 기존)
  ↓
[상단 LegalDraftBanner 렌더] (IS_DRAFT === true)
  ↓
[MD 본문 렌더] → ?raw import된 MD 문자열 → 타이포 매핑 JSX
  ↓ 사용자가 Escape / X / 바깥 클릭
[Dialog Close] (Radix 기본 동작, 기존)
```

접근성: DialogTitle은 MD의 h1 대신 Radix 자동 title 사용. MD 본문 h1은 `sr-only`로 중복 제거.

### DEBT-03: Locked 좌석 클릭 흐름 (기대 동작 확정)

```
[사용자가 locked 좌석 클릭]
  ↓
[SeatMapViewer.handleClick]
  ↓ state === 'sold' 이면 return (click 차단)
  ↓ 그 외 onSeatClick(seatId) 호출
[BookingPage.handleSeatClick]
  ↓ seatState === 'locked' && !selectedSeatIds.has(seatId)
[toast.info("이미 다른 사용자가 선택한 좌석입니다")]
  ↓ return (로컬 상태 변경 없음)
```

**Sold 좌석 vs Locked 좌석 UX 차이 (D-18 명문화):**

| 상태 | SVG 색 | cursor | 클릭 처리 | parent toast |
|------|-------|--------|----------|-------------|
| `available` | tier color | pointer | 선택 추가 | (해당 없음) |
| `locked` | `#D1D5DB` (gray-300) at opacity 0.6 | `not-allowed` | `onSeatClick` 호출 | info toast |
| `sold` | `#D1D5DB` (gray-300) at opacity 0.6 | `not-allowed` | **미호출** (seat-map-viewer에서 차단) | 없음 |

→ 시각적으로는 locked와 sold가 구별되지 않음 (둘 다 LOCKED_COLOR). 이는 Phase 9 범위 외 (색상 구별은 Phase 12 UX 현대화에서 검토). Phase 9는 **클릭 피드백 차이만** 확정.

### DEBT-04: `paidAt` null 렌더링 흐름

```
[admin이 예매 상세 Dialog 오픈]
  ↓
[useAdminBookingDetail → paymentInfo.paidAt: string | null]
  ↓
[formatDateTime(paidAt) → null이면 "—" 반환]
  ↓
[InfoRow "결제일시" 우측 값에 "—" 표시]
```

조건부 렌더링 (`booking.paymentInfo && ...`)은 유지 — paymentInfo 자체가 null이면 "결제수단/결제일시" 행 자체가 숨김.

## Implementation Notes for Planner/Executor

### MD import 방식 (D-06 구체화)
- Next.js 16 + Turbopack에서 `import terms from '@/content/legal/terms-of-service.md?raw'`는 기본 지원 — `next.config.ts`에 별도 webpack loader 불필요.
- 만약 `?raw` import가 타입 오류 발생 시 `apps/web/env.d.ts` (또는 유사 types 파일)에 `declare module '*.md?raw' { const content: string; export default content; }` 추가.

### markdown-to-JSX 라이브러리 선정 (Discretion)
- **Option A (권장):** `react-markdown` + `remark-gfm` — 이미 React 19 호환, 유지보수 활발. 번들 ~30kb. `components` prop으로 클래스 주입 간단.
- **Option B:** 수동 정규식 파서 (h1/h2/p/ul/li만) — 번들 0 증가. 단, 엣지케이스(escaped chars, nested list) 약함. MD 문안이 정적이면 충분.
- **Option C:** `marked` 등 plain-HTML 출력기 + `dangerouslySetInnerHTML` — XSS 주의. Phase 9 법률 문안은 내부 제어이므로 사용 가능하나 A, B 대비 이점 없음.
- planner는 MD 문안 복잡도 확인 후 선택. 선택 시 README/plan에 이유 명시.

### `formatDateTime` 통합 범위 (D-20 구체화)
