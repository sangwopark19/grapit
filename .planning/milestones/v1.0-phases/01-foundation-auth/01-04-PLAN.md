    **OAuth strategies:**

    Each strategy follows the same pattern (PassportStrategy mixin):
    1. Constructor reads client ID, secret, callback URL from ConfigService
    2. validate() callback receives (accessToken, refreshToken, profile, done)
    3. Extracts standardized profile: { provider, providerId, email, name }
    4. Calls done(null, profile) -- does NOT create user here, AuthService handles that

    `kakao.strategy.ts`:
    - PassportStrategy(KakaoStrategy, 'kakao')
    - Config: KAKAO_CLIENT_ID, KAKAO_CLIENT_SECRET, callbackURL from KAKAO_CALLBACK_URL env
    - Profile extraction: profile.id (providerId), profile._json.kakao_account.email, profile.displayName or profile._json.properties.nickname

    `naver.strategy.ts`:
    - PassportStrategy(NaverStrategy, 'naver') using passport-naver-v2
    - Config: NAVER_CLIENT_ID, NAVER_CLIENT_SECRET, callbackURL from NAVER_CALLBACK_URL env
    - Profile extraction: profile.id, profile.emails[0].value, profile.displayName

    `google.strategy.ts`:
    - PassportStrategy(GoogleStrategy, 'google')
    - Config: GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, callbackURL from GOOGLE_CALLBACK_URL env
    - Scope: ['email', 'profile']
    - Profile extraction: profile.id, profile.emails[0].value, profile.displayName

    **Individual strategy spec files:**
    Create `kakao.strategy.spec.ts`, `naver.strategy.spec.ts`, `google.strategy.spec.ts` in the strategies/ directory. Each spec tests the validate() method with mocked profile data to ensure correct extraction of { provider, providerId, email, name }.

    **Social auth guard (guards/social-auth.guard.ts):**
    Generic guard factory that handles the OAuth redirect + callback pattern for any provider.

    **New AuthService methods:**

    `findOrCreateSocialUser(profile: { provider, providerId, email?, name? })`:
    1. Look up social_accounts by (provider, providerId)
    2. If found: user already registered. Load user, generate JWT tokens, return { status: 'authenticated', accessToken, user }
    3. If not found, check if email exists in users table:
       a. If email matches existing user: this is account linking. Still need terms + additional info per D-04.
    4. Generate registrationToken: JWT with payload { provider, providerId, email, name, purpose: 'social-registration' }, expiry 30 minutes.
    5. Return { status: 'needs_registration', registrationToken, socialProfile: { provider, providerId, email, name } }

    `completeSocialRegistration(registrationToken: string, dto: SocialRegisterBody)`:
    SocialRegisterBody contains: name, gender, country, birthDate, phone, termsOfService, privacyPolicy, marketingConsent (same as normal register but no password).
    1. Verify registrationToken JWT, extract { provider, providerId, email }
    2. If expired, throw UnauthorizedException
    3. Check if user with that email already exists:
       a. If exists: create social_account link to existing user, update any missing profile fields, create terms_agreement
       b. If not: create new user (passwordHash = null for social-only), create social_account, create terms_agreement
    4. Generate JWT tokens (same as login)
    5. Return AuthResponse

    **New AuthController endpoints:**

    For each provider (kakao, naver, google):
    - `@Public() @Get('social/:provider')` -- triggers OAuth redirect via AuthGuard(provider)
    - `@Public() @Get('social/:provider/callback')` -- OAuth callback. Calls findOrCreateSocialUser with req.user (profile from strategy). Based on result:
      - If authenticated: redirect to `${FRONTEND_URL}/auth/callback?accessToken=${accessToken}` (frontend reads token and stores)
      - If needs_registration: redirect to `${FRONTEND_URL}/auth/callback?registrationToken=${registrationToken}&status=needs_registration`

    - `@Public() @Post('social/complete-registration')` -- accepts registrationToken + additional info DTO. Returns AuthResponse.

    **SMS Service (apps/api/src/modules/sms/sms.service.ts):**

    Inject ConfigService. Read TWILIO_ACCOUNT_SID, TWILIO_AUTH_TOKEN, TWILIO_VERIFY_SERVICE_SID from env.

    Install Twilio SDK: `pnpm --filter @grapit/api add twilio`

    **Dev mock mode:** If TWILIO_ACCOUNT_SID is empty or NODE_ENV is 'development' and TWILIO_ACCOUNT_SID is not set, use mock mode:
    - sendVerificationCode: log "DEV SMS: code 000000 sent to {phone}" and return success
    - verifyCode: accept '000000' as valid code, reject everything else
    This allows development without a Twilio account.

    **Production mode (Twilio present):**

    `sendVerificationCode(phone: string)`:
    1. Normalize phone to E.164 format: if starts with "010", prepend "+82" and remove leading "0" -> "+8210..."
    2. Call twilio.verify.v2.services(VERIFY_SID).verifications.create({ to: phone, channel: 'sms' })
    3. Return { success: true, message: '인증번호가 발송되었습니다' }
    4. On error: throw BadRequestException with user-friendly message

    `verifyCode(phone: string, code: string)`:
    1. Normalize phone to E.164
    2. Call twilio.verify.v2.services(VERIFY_SID).verificationChecks.create({ to: phone, code })
    3. If status === 'approved': return { verified: true }
    4. If status !== 'approved': return { verified: false, message: '인증번호가 일치하지 않습니다' }

    **SMS Controller (apps/api/src/modules/sms/sms.controller.ts):**

    @Controller('api/v1/sms')

    1. `@Public() @Post('send-code')` sendCode(@Body() { phone }):
       - Validate phone with zod: z.object({ phone: z.string().regex(/^01[016789]\d{7,8}$/) })
       - Call smsService.sendVerificationCode(phone)
       - Return result

    2. `@Public() @Post('verify-code')` verifyCode(@Body() { phone, code }):
       - Validate with zod: z.object({ phone: z.string(), code: z.string().length(6) })
       - Call smsService.verifyCode(phone, code)
       - Return result
