# 인프라 의사결정 제안서

> **프로젝트**: Grapit 티켓 예매 플랫폼
> **작성일**: 2026-03-26
> **목적**: 백엔드 인프라 플랫폼 선정을 위한 의사결정 자료

---

## 1. 배경

현재 아키텍처 설계에서 백엔드 인프라로 **Railway**를 선정했으나, 조사 결과 Railway의 아시아 리전이 **싱가포르 단일**임이 확인되었습니다. 한국 사용자 대상 티켓 예매 플랫폼에서 이것이 어떤 영향을 미치는지 분석하고, 대안으로 **Google Cloud Run (서울 리전)**과 비교하여 최적의 선택지를 제안합니다.

---

## 2. 핵심 쟁점: 서버 위치가 왜 중요한가

티켓 예매 플랫폼은 일반 웹서비스와 다릅니다.

**인기 공연 티켓 오픈 시**, 수천~수만 명의 사용자가 동시에 같은 좌석을 두고 경쟁합니다. 이때 서버와 사용자 간 통신 속도(레이턴시)가 곧 **좌석 확보 성공률**에 직결됩니다.

| 서버 위치 | 한국 사용자까지 왕복 시간 | 예매 전체 플로우 (8회 통신) |
|:---------:|:------------------------:|:--------------------------:|
| **서울** | **2~5ms** | **~300ms** |
| **싱가포르** | **67~80ms** | **~860ms** |

> 같은 좌석을 두 사용자가 동시에 클릭했을 때,
> 서울 서버 사용자의 요청이 **130ms 먼저** 도착합니다.
> 이 차이는 CDN이나 캐싱으로 **해소할 수 없습니다.**

---

## 3. 두 가지 선택지 비교

### 한눈에 보기

|  | Railway (싱가포르) | Cloud Run (서울) |
|--|:------------------:|:----------------:|
| **응답 속도** | 느림 (67~80ms 추가) | 빠름 (2~5ms) |
| **개발 편의성** | 매우 높음 | 보통 (초기 학습 필요) |
| **WebSocket (실시간)** | 10,000 연결 제한 | 제한 없음 (자동 확장) |
| **자동 확장** | 수동 | 자동 |
| **안정성 보증 (SLA)** | 없음 | 99.95% (위반 시 보상) |
| **MVP 월 비용** | ~$24 | ~$26~87 |
| **성장기 월 비용** | ~$71 | ~$160 |

---

### Option A: Railway (싱가포르)

**장점**
- 개발~배포까지 **10분** 만에 가능 (코드 push만 하면 자동 배포)
- 추가 학습 없이 즉시 사용 가능
- 통합 대시보드에서 DB, Redis, 서비스 모두 관리
- MVP 단계에서 **비용이 저렴** ($24/월~)

**단점**
- 서울 리전 없음 → 모든 API 요청에 **+67~80ms 지연**
- WebSocket 동시 연결 **10,000개 제한** → 대형 공연 티켓 오픈 시 부족
- 수평 확장(서버 추가)이 수동
- 공식 SLA 없음, 2025년 DB 장애 3회 발생

**적합한 경우**: 빠르게 MVP를 출시해 시장 반응을 확인하고 싶을 때

---

### Option B: Google Cloud Run (서울)

**장점**
- 서울 리전 → 한국 사용자에게 **최소 레이턴시 (2~5ms)**
- WebSocket 제한 없음 → 50,000+ 동시접속 처리 가능
- 트래픽에 따라 **자동 확장/축소** (티켓 오픈 시 자동 대응)
- 99.95% SLA (위반 시 재정 보상)
- 카나리 배포, 트래픽 분할 등 **무중단 배포 전략** 지원

**단점**
- 초기 설정에 **1~2시간** 소요 (GCP 서비스 10여 개 학습 필요)
- Cloud SQL(PostgreSQL)이 Railway 대비 비쌈 ($12~56/월 vs $6~23/월)
- 성장기 비용이 Railway의 약 **2배** ($160/월 vs $71/월)

**적합한 경우**: 실제 서비스 런칭을 목표로, 안정적이고 빠른 인프라가 필요할 때

## 4. 비용 시나리오

### 4.1 월간 운영 비용

| 단계 | Railway | Cloud Run | 비고 |
|:----:|--------:|----------:|:-----|
| **MVP** (1K DAU) | **$24** | $26~87 | Cloud Run은 cold start 방지 시 $87 |
| **성장기** (10K DAU) | **$71** | $160 | Cloud SQL이 비용 차이의 주 원인 |
| **대규모** (50K DAU+) | $150+ | $200~340 | Railway는 WebSocket 한계 도달 |

### 4.2 티켓 오픈 이벤트 추가 비용 (50K 동시접속, 30분)

| | Railway | Cloud Run |
|--|--------:|----------:|
| 추가 비용 | $5~10 | $70~90 |
| 50K WebSocket 처리 | **불가능** (10K 제한) | **가능** |

## 5. 제안 전략 (3가지)

### 전략 1: Railway로 시작 → Cloud Run으로 이전

```
Phase 1 (MVP, 0~3개월)     → Railway     ~$24/월
Phase 2 (프로덕션, 3개월~)  → Cloud Run   ~$87~200/월
```

| 항목 | 내용 |
|------|------|
| 장점 | MVP를 가장 빠르게 출시, 시장 검증 후 투자 |
| 단점 | 마이그레이션 비용 발생 (1~2일 작업) |
| 리스크 | Railway 종속 기능 사용 시 이전 복잡도 증가 |
| 전제조건 | 처음부터 Dockerfile 사용, Railway 고유 기능 회피 |

### 전략 2: 처음부터 Cloud Run

```
전 기간                     → Cloud Run   ~$26~200/월
```

| 항목 | 내용 |
|------|------|
| 장점 | 마이그레이션 불필요, 서울 레이턴시 처음부터 확보 |
| 단점 | 초기 1~2주 GCP 학습 투자 필요 |
| 리스크 | 개발 초기 속도가 다소 느림 |
| 전제조건 | GCP 기본 학습 (Cloud Run, Cloud SQL, IAM 등) |

### 전략 3: 하이브리드 (Vercel + Cloud Run)

```
프론트엔드 (Next.js)        → Vercel      ~$20/월
백엔드 (NestJS + DB)        → Cloud Run   ~$30~100/월
```

| 항목 | 내용 |
|------|------|
| 장점 | Next.js에 최적화된 Vercel DX + 서울 백엔드 |
| 단점 | 플랫폼 2개 관리, 비용 이중 발생 |
| 리스크 | 1인 개발에서 관리 포인트 증가 |

---

## 6. 권장안

### 전략 2: 처음부터 Cloud Run을 권장합니다

**이유:**

1. **서비스 특성상 레이턴시가 핵심입니다.**
   - 티켓 예매는 "속도 경쟁"이 본질인 서비스입니다
   - 경쟁사(인터파크 등)는 국내 서버를 사용하므로, 싱가포르 서버는 태생적 불리

2. **마이그레이션 비용을 아낄 수 있습니다.**
   - 전략 1은 결국 Cloud Run으로 이전해야 하므로, 처음부터 시작하는 것이 총 비용 절감
   - GCP 학습은 1~2주면 충분하고, 이후 배포 플로우는 Railway와 동일 (`git push` → 자동 배포)

3. **성장에 대비할 수 있습니다.**
   - 인기 공연 티켓 오픈 시 자동 확장
   - WebSocket 동시 연결 제한 없음
   - 99.95% SLA로 결제 안정성 보장

**초기 투자**: GCP 학습 1~2주
**월 비용**: MVP $26~87 → 성장기 $160 → 대규모 $200~340
